## 1. System Classification

**System Type:** Hybrid (CV detection pipeline + direct LLM)

**Description:**
VISION Pack AI extends the existing AI Preprocessor (Python FastAPI microservice) with human-only motion detection (filters animals, vegetation, shadows, weather via refined YOLOv12 + supervision pipeline) and basic facial recognition (whitelist-based, max 50 faces, manual upload via dashboard and mobile app). The system runs fully offline on client hardware (CPU-only), processing live camera frames from ONVIF/RTSP streams. Security operators use the system on the local network — "good" means reliable human-only alerts with zero false positives from non-human motion, accurate face matching at whitelist scale (<50 faces), and sub-500ms per-frame latency on consumer CPU hardware.

**Critical Failure Modes:**
1. **False positive alerts from non-human motion** — animals, vegetation, shadows, weather triggering alarms, eroding operator trust and causing alert fatigue (the #1 reason security systems get ignored)
2. **False negatives — missed human detection** — an actual person entering a zone goes undetected, creating a genuine security gap
3. **Face match errors** — whitelisted person not recognized (false negative = access denied, alerts fired for authorized personnel) or stranger recognized as whitelisted (false positive = security bypass)
4. **Zone filtering failures** — detection zones not respected; person outside configured zone triggers alert, or person inside zone is ignored
5. **Latency spikes blocking real-time operation** — frame processing exceeds 500ms on CPU, causing stream backlog and delayed alerts

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Physical security / video surveillance (SMB to enterprise, Africa-focused)

**User Population:** Security operators (guards, facility managers, business owners) using the system on local network. Non-technical users — installation may be done by a local integrator. French-speaking (Niger/West Africa). Typical technical comfort: can use a mobile app and web dashboard, but not configure networks.

**Stakes Level:** High — false alerts erode trust and cause operators to ignore the system entirely. Missed detections create genuine security vulnerabilities (theft, intrusion, unauthorized access).

**Output Consequence:** AI detection output determines whether a human operator is alerted (push notification, SMS, WhatsApp). False positive = operator distracted, false negative = breach undetected. Face recognition output controls whitelist access decisions — false negative denies entry to authorized personnel, false positive grants access to unauthorized person.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|---------------------|--------|--------|
| Human detection accuracy | Person detected in zone within 2s; no alerts from animals, curtains, trees, headlights | "Trois alertes par heure pour un chat" / "La climatisation déclenche une alarme" | High — operators will mute/disconnect the system if noise >1 false positive per day | West African security integrator interviews |
| Face recognition (whitelist) | Known employee recognized at 95%+ consistency across lighting conditions | Employee stands 10s at door, not recognized (false negative) | High — productivity loss + embarrassment at access point | Physical security practitioner reports |
| Face rejection (stranger) | Unknown person correctly flagged as "not recognized" | Stranger matched to wrong employee profile (false positive) | Critical — unauthorized access granted | Security industry standards (access control) |
| Zone compliance | Detection fires only within configured polygon/rect zones | Detection fires outside zone boundaries | Medium — reduces sensitivity configurability value | Product spec (VIS-11) |
| CPU latency | Frame processed in <500ms on Intel NUC / similar consumer hardware | Processing queue grows >5s behind real-time | Medium — delayed alerts miss live events | Hardware constraint (no GPU in VISION tier) |

### Known Failure Modes in This Domain

1. **Animal/vegetation triggers on outdoor cameras** — most common source of false alerts in African deployments (goats, dogs, birds, moving tree branches). Standard motion sensors unusable without AI filtering.
2. **Lighting change flooding** — clouds, sunrise/sunset, vehicle headlights cause pixel-level change that naive motion detection flags as "movement"
3. **Rain/weather artifacts** — rain streaks, fog, dust on lens triggering false positives
4. **Shadow cast by passing vehicles** — shadow moves across frame, detected as moving person
5. **Offline camera → no AI processing** — if the camera stream drops, the AI pipeline silently produces no output (no alert) even if an event occurred during the gap

### Regulatory / Compliance Context

- **HAPDP (Niger data protection authority)** — biometric data (face embeddings) constitutes sensitive personal data under HAPDP. Requires:
  - Declared processing register entry (BAS-30, Phase 4 scope)
  - Subject access right (see/rectify/delete) — Phase 4
  - Pseudonymization recommended — embeddings stored without plaintext name linkage
- **HAPDP compliance for VISION tier**: Face embeddings stored locally, no cloud transmission; whitelist only (no surveillance/unconsented identification). VISION tier is within HAPDP exemption for personal household security systems if no third-party access.
- **General surveillance regulations**: Camera consent signage (VIS-32) is a Phase 4 requirement. VISION tier should default to private home/small business usage pattern.

## 2. Framework Decision

**Selected Framework:** Extend existing stack (ultralytics YOLOv12 + supervision + FastAPI + insightface)

**Version:**
- ultralytics >=8.4
- supervision >=0.25
- insightface >=0.7.3
- onnxruntime >=1.17.0
- fastapi 0.115.0 (existing)

**Rationale:**
Phase 2 VISION AI work is computer vision (YOLO detection filtering, face recognition, zone logic), not LLM orchestration. The existing stack — raw ultralytics + supervision in the FastAPI microservice for CV, and direct Ollama integration in the custom NestJS AiAgentModule for LLM — is already correct. Adding LangChain, LangGraph, or any third-party LLM framework would violate the project constraint against framework rewrites and add unnecessary abstraction to a working pipeline. For VIS-07 facial recognition, insightface with onnxruntime CPU inference is the best fit — lightweight, offline-capable, Apache 2.0 licensed, and trivially extensible as a new endpoint in services/ai-preprocessor/.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain / LangGraph | Phase 2 VISION is CV pipeline, not LLM orchestration; adding framework violates "no framework rewrite" constraint |
| DeepFace | Heavier dependency, slower CPU inference than insightface, wraps multiple models unnecessarily for 50-face whitelist |
| face_recognition (dlib) | GPL-licensed (license conflict with commercial SaaS), slower CPU inference, no built-in detection model |
| MediaPipe Face | No built-in recognition (detection only), would need separate encoder for embeddings |

**Vendor Lock-In Accepted:** No — all selected libraries are open-source, model-agnostic, and self-hosted.

# 4. Temporal smoothing — only alert if detected for N consecutive frames

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `ultralytics.YOLO` | PyTorch-based object detection model. Loaded once as a lazy singleton. `model(frame, conf=0.45)` returns list of `Results`. | Every frame: run detection, then convert to `sv.Detections`. |
| `sv.Detections` | Supervision's unified detection container. Holds `xyxy`, `confidence`, `class_id`, `tracker_id`, `mask` as parallel numpy arrays. Index with boolean masks for filtering. | After YOLO inference: filter by class, confidence, zone; pass to ByteTrack; extract bbox for face crop. |
| `sv.PolygonZone` | Spatial filter defined by a polygon of (N, 2) vertices. `zone.trigger(detections)` returns a boolean mask — `True` where bbox centre is inside the polygon. | Geo-fencing: only alert if a person is detected inside a configured zone (e.g., restricted area). |
| `insightface.FaceAnalysis` | Face detection + recognition pipeline. Wraps SCRFD detection + ArcFace recognition in one `app.get(frame)` call. Returns list of `Face` objects with `bbox`, `kps`, `det_score`, `normed_embedding`. | Every frame where a person is detected: crop+align face, extract embedding, compare against whitelist. |
| `face.normed_embedding` | L2-normalised 512-d embedding vector. Cosine similarity = dot product (since both vectors are unit-length). | Compare against stored whitelist embeddings to identify or reject a person. |

### Common Pitfalls

1. **insightface crashes with CUDA errors on CPU-only hardware.**
   `FaceAnalysis()` defaults to `['CUDAExecutionProvider', 'CPUExecutionProvider']`. If CUDA libraries are absent (or onnxruntime-gpu is not installed), it falls back silently on some systems but crashes with `onnxruntime.capi.onnxruntime_pybind11_state.Fail` on others. **Fix:** Always pass `providers=["CPUExecutionProvider"]` and `ctx_id=-1` explicitly for VISION's CPU-only deployment.

2. **YOLO model is not thread-safe.**
   The underlying PyTorch model holds state; calling `model(frame)` from two threads simultaneously can produce corrupted results or segfaults. **Fix:** Use one model instance per thread/process, or wrap calls with `ThreadingLocked` decorator. In the FastAPI async context, run YOLO inference via `loop.run_in_executor(None, model.predict, frame)` to avoid blocking AND thread-safety issues.

3. **insightface `app.get()` re-downloads the model on every call in some Docker environments.**
   The `~/.insightface/models/buffalo_l/` directory must be persisted across container restarts or pre-loaded in the Docker image. If the directory is writable but empty, `app.get()` silently re-downloads the 180 MB model pack on every container start, causing 30-60s cold-start latency. **Fix:** Pre-download in Dockerfile with `python -c "from insightface.app import FaceAnalysis; FaceAnalysis(name='buffalo_l', providers=['CPUExecutionProvider'])"` during build.

4. **supervision `sv.PolygonZone` polygon scaling mismatch.**
   `PolygonZone` accepts polygon coordinates in pixel space, and `frame_resolution_wh` must match the actual frame dimensions. If the frame is resized before detection (e.g., YOLO resizes to 640×640 internally), the polygon coordinates must be in the *original* (un-resized) frame space. **Fix:** Store zone polygons relative to the camera's native resolution (from ONVIF/RTSP metadata), and scale them to each frame's actual dimensions before constructing the zone.

5. **ONNX Runtime session memory leak in long-running containers.**
   Each `onnxruntime.InferenceSession` allocates memory that is not freed when the session goes out of scope in Python. insightface creates a new session internally. On a CPU-only system processing 24/7, this can cause 5-10 MB/hour memory growth. **Fix:** Use a single persistent `FaceAnalysis` instance (already singleton), and periodically monitor RSS with `psutil` in the health-check endpoint.

## 4. Implementation Guidance

### Model Configuration

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Detection model | `yolov12n.pt` (nano) | YOLOv12n has ~3.2M params, ~80ms CPU inference at 640×640 per ultralytics benchmarks. Nano variant is the only choice for <500ms target on consumer CPU. |
| Detection confidence | `0.45` (configurable per-camera via `DETECTION_CONFIDENCE`) | Tuned to balance recall (don't miss small/distant persons) vs. precision (don't detect animals/vegetation as person). Cameras with more false positives (outdoor, vegetation) should use 0.50–0.55. |
| NMS IoU threshold | `0.50` (YOLO default) | Standard value; reduces duplicate boxes around the same person. |
| Input resolution | `640×640` (YOLO internal) | Fixed resize — the pipeline receives the camera's native resolution frame, YOLO resizes internally. Zone polygons must be in **native** frame coordinates. |
| Face detection model | `buffalo_l` pack (SCRFD + ArcFace) | InsightFace's default model pack provides SCRFD face detection (retinaface-style) + ArcFace recognition. ~100-300ms CPU per frame with both detection + recognition. |
| Face match threshold | `0.48` (configurable: `FACE_MATCH_THRESHOLD`) | Cosine similarity threshold. 0.48 is tight enough to reject strangers but lenient enough for lighting variation. Tune per camera if lighting conditions differ significantly. |
| Max whitelist faces | 50 (enforced by license feature gate) | VISION tier limit hard-coded via `@RequiresFeature('vision_face_whitelist')` in NestJS. BASTION tier removes this cap. |
| Temporal smoothing window | 5 frames (~1s at 5 FPS ingest) | Prevents flickering alerts. A person must be detected in ≥ 3/5 consecutive frames before an alert fires. |

### Core Pattern

The complete VISION frame pipeline flows through these stages. The implementation lives primarily in existing endpoints with additions for face recognition.

```python
import asyncio
import time
import numpy as np
from PIL import Image

# ── Existing imports (from app/models/detector.py, app/models/tracker.py) ──
from app.models.face_recogniser import get_face_recogniser

# ── Temporal smoothing buffer (per camera, in-memory) ──
# Keyed by (camera_id, tracker_id) → list of detection booleans for last N frames
_frame_buffer: dict[tuple[str, int], list[bool]] = {}
MAX_SMOOTHING_FRAMES = 5
MIN_POSITIVE_FRAMES = 3


async def process_camera_frame(
    camera_id: str,
    org_id: str,
    frame: np.ndarray,           # RGB, native resolution
    detection_zones: list[np.ndarray],   # polygon vertices in native coordinates
) -> list[dict]:
    """Run the full VISION detection + face recognition pipeline on one frame.

    This function is called by the existing /detect endpoint for every frame
    that passes the pre-filter (motion trigger or periodic ingest tick).
    """
    start = time.time()
    results: list[dict] = []

    # ── 1. YOLO human-only detection ──────────────────────────────────
    # Run in executor to avoid blocking the async event loop
    loop = asyncio.get_running_loop()
    sv_detections = await loop.run_in_executor(
        None, lambda: _run_yolo_detection(frame),
    )

    if len(sv_detections) == 0:
        return []

    # ── 2. Zone filtering (polygon geofencing) ───────────────────────
    # Each camera can have 0+ zones; a detection triggers an alert if
    # it falls inside ANY active zone (or unconditionally if no zones configured)
    in_any_zone = np.zeros(len(sv_detections), dtype=bool)
    for polygon in detection_zones:
        zone = sv.PolygonZone(
            polygon=polygon,
            frame_resolution_wh=(frame.shape[1], frame.shape[0]),
        )
        in_any_zone |= zone.trigger(detections=sv_detections)

    # If zones are configured, a person outside all zones is silently ignored
    if len(detection_zones) > 0:
        sv_detections = sv_detections[in_any_zone]

    if len(sv_detections) == 0:
        return []

    # ── 3. Temporal smoothing (flicker suppression) ──────────────────
    # Only keep detections whose tracker_id has been seen in ≥ 3 of the
    # last 5 frames. New tracker_ids (first frame) are buffered but not alerted.
    stable_detections = []
    for i in range(len(sv_detections)):
        tracker_id = int(sv_detections.tracker_id[i]) if sv_detections.tracker_id[i] is not None else None
        if tracker_id is not None:
            key = (camera_id, tracker_id)
            if key not in _frame_buffer:
                _frame_buffer[key] = []
            buf = _frame_buffer[key]
            buf.append(True)
            if len(buf) > MAX_SMOOTHING_FRAMES:
                buf.pop(0)
            if sum(buf) >= MIN_POSITIVE_FRAMES:
                stable_detections.append(i)
        else:
            # No tracker ID (tracker not initialised yet) → include anyway
            stable_detections.append(i)

    if len(stable_detections) == 0:
        return []   # not yet stable

    sv_detections = sv_detections[stable_detections]

    # ── 4. Face recognition (if person and face is visible) ──────────
    # Crop each detected person's bounding box, run insightface on the crop.
    # This avoids running face detection on the full frame multiple times.
    face_recogniser = get_face_recogniser()   # lazy singleton, CPU-only

    for i in range(len(sv_detections)):
        x1, y1, x2, y2 = map(int, sv_detections.xyxy[i])
        # Expand bbox 20% for better face capture (head is near top of bbox)
        y1 = max(0, y1 - int((y2 - y1) * 0.2))
        person_crop = frame[y1:y2, x1:x2]

        if person_crop.size == 0:
            continue

        # insightface expects BGR input
        crop_bgr = cv2.cvtColor(person_crop, cv2.COLOR_RGB2BGR)
        faces = face_recogniser.get(crop_bgr)

        face_match = None
        for face in faces:
            emb = face.normed_embedding
            best_sim, best_id = _match_whitelist(emb)
            if best_sim >= FACE_MATCH_THRESHOLD:
                face_match = {"face_id": best_id, "similarity": round(float(best_sim), 3)}
                break   # take highest-confidence face in the crop

        # ── 5. Build alert payload ────────────────────────────────────
        # Continue even if no face match (unknown person alert)
        results.append({
            "bbox": [x1, y1, x2, y2],
            "confidence": float(sv_detections.confidence[i]),
            "tracker_id": int(sv_detections.tracker_id[i]) if sv_detections.tracker_id[i] is not None else None,
            "face": face_match,        # None if no face / not recognised
            "in_zone": True,
        })

    # Clean stale tracker IDs from buffer (>30s no updates handled by a periodic task)
    _prune_stale_buffers()

    logger.info(
        "Frame processed: %dms, %d human detections, %d face matches",
        int((time.time() - start) * 1000),
        len(results),
        sum(1 for r in results if r["face"] is not None),
    )
    return results


# ── Helper functions ─────────────────────────────────────────────────────────

def _run_yolo_detection(frame: np.ndarray) -> sv.Detections:
    """YOLO inference wrapper — runs in executor to avoid GIL blocking."""
    from app.models.detector import detect
    return detect(frame, confidence=0.45)

def _match_whitelist(embedding: np.ndarray) -> tuple[float, str | None]:
    """Return (best_similarity, face_id) against the in-memory whitelist cache."""
    best_sim = -1.0
    best_id = None
    for entry in _whitelist_cache:   # loaded at startup, refreshed periodically
        sim = float(np.dot(embedding, entry["embedding"]))
        if sim > best_sim:
            best_sim = sim
            best_id = entry["id"]
    return best_sim, best_id
```

### State Management

| State | Where Stored | Lifetime | Access Pattern |
|-------|-------------|----------|----------------|
| YOLO model instance | Python global in `app/models/detector.py` (`_yolo_model`) | Container lifetime. Loaded on first request. | Read-only once loaded; thread-safe via `run_in_executor` (each call in its own thread releases GIL). |
| insightface `FaceAnalysis` instance | Python global in `app/models/face_recogniser.py` (`_face_app`) | Container lifetime. Loaded on first `/face/*` request. | Read-only once loaded; `app.get()` is CPU-bound and must be called in executor. |
| Face whitelist embeddings | PostgreSQL (table `face_whitelist` in `apps/api/prisma/schema.prisma`) + in-memory cache (`_whitelist_cache` list of dicts) | Cache refreshes every 60s via periodic task. Database is canonical. | API writes (add/delete faces) → AI preprocessor cache invalidated via webhook or TTL. Cache is a simple list of `{id, name, embedding: np.ndarray}` in Python. |
| Detection zone polygons | PostgreSQL (`camera_zone` table or similar) + in-memory cache keyed by `camera_id` | Per-session, refreshed on camera config change. | Loaded once per camera when detection pipeline starts. Polygons stored as JSON arrays of `[x,y]` vertices. Converted to `np.ndarray` for `sv.PolygonZone`. |
| Temporal smoothing buffer | Python dict `_frame_buffer: dict[tuple[str, int], list[bool]]` | In-memory only; lost on restart (acceptable — smoothing restarts). | Keyed by `(camera_id, tracker_id)`. Each entry is a rolling window of detection booleans. Pruned after 30s of no activity (periodic cleanup task). |

**Whitelist cache refresh flow:**

```python
# In app/models/face_recogniser.py — periodic cache refresh
import httpx

_whitelist_cache: list[dict] = []
_WHITELIST_API_URL = f"{settings.NESTJS_API_URL}/api/internal/face-whitelist"

async def refresh_whitelist_cache() -> None:
    """Fetch the face whitelist for this organisation from NestJS API."""
    global _whitelist_cache
    try:
        async with httpx.AsyncClient(timeout=10) as client:
            resp = await client.get(_WHITELIST_API_URL)
            resp.raise_for_status()
            data = resp.json()  # list of {id, name, embedding_base64}
            embeddings = []
            for entry in data:
                emb_bytes = base64.b64decode(entry["embedding_base64"])
                emb = np.frombuffer(emb_bytes, dtype=np.float32).copy()
                embeddings.append({"id": entry["id"], "name": entry["name"], "embedding": emb})
            _whitelist_cache = embeddings
            logger.info("Whitelist cache refreshed: %d faces", len(embeddings))
    except Exception as e:
        logger.error("Whitelist refresh failed (using stale cache): %s", e)
```

### Context Window Strategy

Not applicable in the LLM sense — this is a CV pipeline, not a language model with a token limit. However, three analogous constraints apply:

1. **Frame rate cap (bandwidth budget).** The CPU can process ~3-5 FPS with YOLOv12n + face recognition. If the camera sends 25 FPS, frames must be skipped or pre-filtered by motion detection. **Strategy:** Always drop intermediate frames — process at most 1 FPS per camera for face recognition (full pipeline), and only YOLO-detection at 5 FPS for motion alerts. Face recognition runs only when YOLO detects a person that passes zone + smoothing filters.

2. **Temporal smoothing window size.** The `_frame_buffer` stores the last 5 detection statuses per `(camera_id, tracker_id)`. This is a hard limit — old frames are dropped automatically. After a tracker ID disappears for 30+ seconds, its buffer is pruned. This prevents unbounded memory growth across thousands of tracker IDs over days of uptime.

3. **Whitelist size limit.** VISION tier enforces max 50 faces. At 512 floats per embedding (512 × 4 bytes = 2 KB), the in-memory cache is ~100 KB — negligible. The dot-product comparison loop over 50 embeddings takes <1 ms per face crop. If BASTION later increases the limit to thousands of faces, an ANN index (e.g., `faiss.IndexFlatIP`) should replace the linear scan.

### Structured Outputs with Pydantic

The existing codebase already follows this pattern with `DetectionRequest` / `DetectionResponse` in `detection.py`. For face recognition, define new models in `app/schemas/face_recognition.py`:

```python
from pydantic import BaseModel, Field
from typing import Optional


class FaceWhitelistEntry(BaseModel):
    """A single known person in the organisation's face whitelist.

    Stored in PostgreSQL. Embedding serialised as base64 float32 array.
    """
    id: str
    name: str
    embedding_base64: str          # np.ndarray → base64 on write
    created_at: str | None = None
    updated_at: str | None = None


class FaceRecognitionRequest(BaseModel):
    """Request to recognise faces in a camera frame."""

    camera_id: str
    image_base64: str
    timestamp: str | None = None
    organization_id: str
    min_face_size: int = 80       # ignore faces smaller than 80×80 px (saves CPU)
    match_threshold: float = 0.48  # same as config default


class FaceMatch(BaseModel):
    """A recognised (or unrecognised) face in the frame."""

    bbox: list[float]              # [x1, y1, x2, y2] in original frame coords
    confidence: float              # face detection confidence
    identity: str | None = None    # whitelisted name if matched, else None
    identity_id: str | None = None # UUID if matched, else None
    similarity: float | None = None# cosine similarity if matched, else None


class FaceRecognitionResponse(BaseModel):
    """Response payload for face recognition results."""

    faces: list[FaceMatch]
    camera_id: str
    processing_time_ms: float
    total_whitelist_size: int      # how many embeddings were compared against


class FaceRegistrationRequest(BaseModel):
    """Register a new face in the whitelist.

    Sent from dashboard/mobile → NestJS API → AI Preprocessor (for embedding extraction).
    """
    name: str = Field(..., min_length=1, max_length=100)
    image_base64: str              # face photo uploaded by admin
    organization_id: str

    # VISION limit enforced by feature gate in NestJS, but validate here too
    # (max 50 faces per org in VISION tier)
```

**Validation rule:** The `FaceRecognitionRequest.match_threshold` must be between 0.20 and 0.95. Values outside this range are clamped silently:

```python
class FaceRecognitionRequest(BaseModel):
    match_threshold: float = 0.48

    @classmethod
    def clamp_threshold(cls, v: float) -> float:
        return max(0.20, min(0.95, v))
```

**No retry logic needed** for the CV pipeline — unlike LLM JSON parsing, the detection + recognition output is deterministic given the same input. Retries apply only to the **notification delivery** to NestJS API (handled by existing `_notify_nestjs()` with a timeout and error log; no retry to avoid duplicate alerts).

### Async-First Design

**FastAPI pattern (already in use):** All endpoints are `async def`. I/O-bound operations (HTTP calls, DB queries via httpx) use `await`. CPU-bound operations (YOLO inference, face recognition, NumPy computations) **must** use `run_in_executor` to avoid blocking the event loop.

**The one common mistake in this pipeline:**
```python
# ❌ WRONG — blocks the entire event loop for 100-300ms
@app.post("/detect")
async def detect_objects(request: DetectionRequest):
    frame = np.array(...)
    results = model(frame)           # CPU-bound, blocks event loop
    # All other requests are stalled until this returns
```

```python
# ✅ CORRECT — runs YOLO in a thread pool, releases event loop
@app.post("/detect")
async def detect_objects(request: DetectionRequest):
    frame = np.array(...)
    loop = asyncio.get_running_loop()
    results = await loop.run_in_executor(None, model.predict, frame)
    # Other requests can be processed while YOLO runs
```

**Stream vs. await:** Not applicable to this CV pipeline — frame responses are <1 KB JSON, not streaming. Video streaming is handled separately via RTSP/go2rtc (VIS-01 to VIS-05). The detection endpoint always returns a complete JSON response.

**Pool configuration:** The default `ThreadPoolExecutor` (used by `run_in_executor(None, ...)`) creates `min(32, os.cpu_count() + 4)` threads. For a CPU-only pipeline, limit this to `os.cpu_count()` to avoid thread oversubscription:

```python
import os
from concurrent.futures import ThreadPoolExecutor

# In app/main.py during startup
_cpu_executor = ThreadPoolExecutor(max_workers=os.cpu_count())

# Then in route handlers:
results = await loop.run_in_executor(_cpu_executor, model.predict, frame)
```

### Prompt Engineering Discipline

**Not directly applicable to the CV detection pipeline** — there is no LLM prompt in the YOLO + insightface flow. The "prompt" is the model architecture and training data.

**For the facial recognition component, the "prompts" are:**
- The **match threshold** (equivalent to system prompt strictness). A lower threshold (e.g., 0.35) is more permissive (fewer false negatives but more false positives). A higher threshold (0.55+) is stricter. **Rule:** Default to 0.48 for general use. Allow per-camera override if the integrator calibrates during setup.
- The **min face size** parameter (equivalent to token budget). Faces smaller than 80×80 px produce low-quality embeddings. The face recognition endpoint silently skips them rather than generating unreliable matches.

**For the LLM branch** (VIS-09, Phase 3 BASTION scope), this section will become relevant. The single principle to carry forward: **separate routing from generation**. The YOLO pipeline determines *if* something happened; the LLM (if used later) would determine *what to do about it*.

### Context Window Management

**Not applicable in the LLM sense** — see §4 Context Window Strategy for the CV-specific equivalents (frame rate cap, temporal smoothing window, whitelist size limit). The three resource budgets that replace "context window" are:

1. **CPU time budget per frame** — max 500ms total. YOLO detection (~80ms) + face recognition on each person crop (~200ms) + overhead (~30ms) = ~310ms for a single-person frame. Two persons visible simultaneously → ~510ms, exceeding budget. **Mitigation:** Process face recognition on at most **one person crop per frame** (the one with highest detection confidence). Lower-priority crops wait for the next frame.

2. **Memory budget for embedding cache** — VISION max 50 faces × 2 KB = 100 KB. At BASTION scale (thousands), use FAISS index for ANN search (sub-millisecond even at 100K embeddings).

3. **Periodic cleanup of stale tracking state** — The `_frame_buffer` grows unbounded if tracker IDs are never pruned. Run a cleanup every 60 seconds:

```python
async def prune_stale_buffers():
    """Remove tracker buffers not seen in 30+ seconds."""
    global _frame_buffer
    now = time.time()
    stale_keys = [k for k, v in _frame_buffer.items()
                  if now - getattr(v, "_last_seen", now) > 30]
    for k in stale_keys:
        del _frame_buffer[k]
    if stale_keys:
        logger.debug("Pruned %d stale tracking buffers", len(stale_keys))
```

### Cost and Latency Budget

**Hardware context:** VISION tier runs on the customer's local hardware — typically an Intel NUC, Mini PC, or re-purposed desktop, CPU-only (no GPU). The customer pays for the hardware once; there is no per-call API cost. The budget is **latency**, not **dollars**.

| Operation | CPU Latency (estimate) | Notes |
|-----------|----------------------|-------|
| YOLOv12n detection (640×640) | 60-100 ms | Nano variant. Scales with CPU generation (NUC i5 vs. i7). |
| supervision Detections conversion | <1 ms | numpy array ops |
| ByteTrack tracking update | <2 ms | Per-target Kalman filter update |
| PolygonZone trigger check | <0.5 ms | NumPy point-in-polygon test |
| insightface face detection (full frame) | 80-200 ms | Buffalo_L SCRFD model |
| insightface face recognition (per crop) | 30-80 ms | Embedding extraction only if crop is >=80×80 |
| Cosine similarity (50-embedding whitelist) | <1 ms | Pure numpy dot product |
| Image decode + convert (base64 → frame) | 10-30 ms | JPEG decode + RGB conversion |

**Cumulative per-frame budget (worst case):**

| Scenario | Components | Total | Within 500ms? |
|----------|-----------|-------|---------------|
| No person in frame | YOLO only | ~100ms | ✅ Yes |
| Person detected, no face | YOLO + ByteTrack + zone + smoothing | ~110ms | ✅ Yes |
| Person + face recognised | YOLO + track + zone + smoothing + face detect + recognise (50 entries) | ~350ms | ✅ Yes |
| Two persons visible simultaneously | YOLO + 2× face recognition | ~510ms | ⚠️ Marginally over — defer 2nd face to next frame |

**Optimisation strategies:**

1. **Skip face recognition if frame-level brightness < threshold** (night/dark frames produce garbage embeddings). Simple heuristic: mean pixel value < 20 → skip face step entirely.
2. **Temporal cooldown per tracker_id:** After a face match fires for a given tracker_id, suppress re-recognition for 5 seconds (the person is already identified; no need to re-process every frame).
3. **Motion pre-filter:** If using continuous processing (not event-triggered), run a cheap frame-differencing step before invoking YOLO. Skip if pixel change < threshold (no motion).

**Caching:**

- **Exact-match cache:** Face embeddings are deterministic for the same input crop. If the exact same crop region is submitted (unlikely in real-time, but possible in tests), cache the result.
- **Semantic cache (embedding-level):** Not practical — every frame is slightly different, so embeddings will differ at floating-point precision. The temporal smoothing buffer serves as the "cache" (don't re-alert for the same tracker_id within the smoothing window).
- **Whitelist cache:** Always in memory (see §4 State Management). Never hit the database for the comparison loop.

### Dimensions

All dimensions are grounded in the domain expert rubric from §1b. Note: this is a **computer vision pipeline**, not an LLM system. LLM judges are not applicable. Measurement approaches are Code-based (deterministic checks) or Human evaluation (subjective quality assessment via labeled dataset).

| **CPU latency** | **PASS:** Per-frame processing <500ms p95 on Intel NUC i5 (or equivalent consumer CPU) for single-person frame. <br>**FAIL:** Processing queue grows >5s behind real-time — delayed alerts miss live events (§1b). <br>**Measurement:** Instrument each frame's processing time; track p50/p95/p99 across 1000-frame sample. | Code (timing instrumentation) | High |

## 6. Guardrails

### Online (Real-Time)

For CV pipelines, online guardrails are lightweight checks that run on every frame before alert generation. Each guardrail must add <10ms overhead.

| Guardrail | Trigger | Intervention | Latency Budget |
|-----------|---------|--------------|---------------|
| **Minimum confidence floor** | YOLO detection with `confidence < 0.35` (hard floor, separate from configurable `DETECTION_CONFIDENCE` of 0.45) | Drop the detection before it enters zone/smoothing pipeline | <1ms (numpy comparison) |
| **Zone filter** (VIS-11) | Detection bbox centre falls outside all configured detection polygons | Silently ignore the detection — no alert, no face recognition | <0.5ms (PolygonZone.trigger) |
| **Dark frame skip (face rec)** | Frame mean pixel value < 20 (night/dark — embeddings are unreliable) | Skip face recognition step entirely; continue YOLO-only alert pipeline | <1ms (numpy mean) |
| **Camera offline detection** | Frame not received from camera for >15s (missing ingest tick) | Suppress all alerts from that camera; log "camera offline — no alerts possible" | N/A (timeout check) |
| **Cooldown per tracker ID** | Tracker ID already triggered a face match in the last 5 seconds | Skip face recognition for this tracker; still fire person-detected alert | <0.5ms (dict lookup) |
| **Max face crops per frame** | More than 1 person crop detected in a single frame | Process only highest-confidence person's face; defer remaining to next frame | <1ms (sort + slice) |
| **Whitelist size gate** | Whitelist cache size exceeds VISION tier limit (50) | Refuse new face registration; return 402 Payment Required with "Upgrade to BASTION" | N/A (API call check) |

### Offline (Flywheel)

Quality signals that feed back into system improvement. Sampled / batched, not real-time.

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **False positive rate (weekly)** | Sample 100% of operator-flagged false alarms per camera. Compute: `FP_rate = flagged_false_alarms / total_alerts_per_camera_per_week`. | If FP_rate > 1 false alarm per camera per day (7/week): increase `DETECTION_CONFIDENCE` by 0.05 for that camera and notify operator to review zone/sensitivity settings. |
| **False negative rate (monthly)** | Run full labeled test set (§5) against current detection pipeline once per month. Compare `recall` against baseline. | If recall drops >3% from baseline: flag for integrator review. Possible causes: model weight corruption, lighting condition shift, camera angle drift. |
| **Face match consistency (monthly)** | Run whitelisted-face test set (§5) through face recognition pipeline. Compute `TPR@threshold` for each whitelist member. | If any whitelist member's TPR drops below 80%: flag for manual review (poor enrolment photo? lighting change at access point?). If aggregate TPR drops >5%: consider recalibrating `FACE_MATCH_THRESHOLD` for that camera. |
| **Unknown person rejection rate (weekly)** | Log all "unknown face" events per camera per week. Compare count vs. known-person matches (proxy for access point busy-ness). | If unknown-person rejection rate < 99.5% (i.e., >0.5% of strangers matched as known): emergency threshold tightening — increase `FACE_MATCH_THRESHOLD` by 0.05 and alert security integrator. |
| **Latency p99 drift (daily)** | Instrument per-frame processing time for all frames. Compute p99 latency per camera per day. | If p99 latency > 1000ms for any camera: throttle frame rate for that camera (skip 1 in 2 frames). If sustained >2000ms: disable face recognition on that camera (YOLO-only). |
| **Detection queue depth (real-time → weekly trend)** | Monitor BullMQ queue depth of pending frames awaiting AI processing. | If queue depth exceeds 50 frames: increase frame skip ratio. If queue never drains (sustained growth): recommend operator reduce camera count or upgrade hardware (NUC → better CPU). |

### Smart Sampling Strategy

For the offline flywheel, we cannot store every frame (bandwidth/storage limited on customer hardware). Instead, use signal-weighted sampling to retain the most informative frames:

| Sample Category | Sampling Rate | Retained Where | Rationale |
|----------------|---------------|----------------|-----------|
| **Low-confidence detections** (`confidence < 0.60`) | 100% | `logs/ai-preprocessor/sampled/` (JPEG, resized 640×640) | These are the ambiguous cases most likely to be FP or FN errors. Reviewing them directly improves model calibration. |
| **Zone boundary detections** (bbox centre within 20px of zone polygon edge) | 100% | Same as above | Zone edge cases where a person might be erroneously included or excluded due to polygon precision. |
| **Crowded frames** (>3 detections) | 100% | Same as above | Stress test for face rec pipeline — likely to expose tracking or crop prioritisation failures. |
| **Operator-flagged false alarms** | 100% + all surrounding frames (±2s) | `detection_id` stored in alert DB with `flagged=true`; frames retained in alert video clip | Highest-value signal — every false alarm is a training opportunity. |
| **High-confidence detections** (`confidence > 0.85`, single person, clear face match) | 5% (random) | Same as above | Baseline confidence check — verify that "easy" cases aren't silently degrading over time. |
| **Face match near threshold** (similarity within ±0.05 of `FACE_MATCH_THRESHOLD`) | 100% | Same as above | Threshold boundary cases — small lighting changes could flip the match decision. Calibrating these is critical. |

**Sampled frame retention policy:** Maximum 1000 sampled frames per camera. Oldest frames evicted when limit reached. Frames retained for 30 days maximum.

**Review workflow:** Sampled frames are visible in the dashboard "AI Quality" section (Phase 4 scope, baseline in Phase 2). Operator or integrator can scroll through, confirm correct/incorrect, and their feedback is stored as additional ground truth for the next model calibration pass.
