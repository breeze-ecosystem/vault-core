  <action>
    Create RecordingModule with recording pipeline, retention enforcement, and clip export:

    1. Create `recording.module.ts` — import ScheduleModule for @Cron, standard @Module
    2. Create `recording.service.ts`:
       - `activeRecordings: Map<string, ChildProcess>` — tracks FFmpeg processes per camera
       - `startRecording(cameraId, rtspUrl, codec = 'h264')` — spawns FFmpeg subprocess:
         ```
         ffmpeg -rtsp_transport tcp -i {rtspUrl} -c {h264_or_h265} -f hls -hls_time 60 -hls_list_size 0 -strftime 1 /mnt/recordings/{cameraId}/{date}/segment_%Y%m%d_%H%M%S.ts
         ```
         Creates output dir structure: `/mnt/recordings/{cameraId}/{YYYY-MM-DD}/`
       - `stopRecording(cameraId)` — kills FFmpeg process, cleans up
       - `getRecordingStatus(cameraId)` — returns { isRecording, codec, storagePath, estimatedSize }
       - `getConfig(orgId)` → RecordingConfig
       - `updateConfig(orgId, data)` → update retention/codec via Prisma
       - On module init: read camera configs and start recordings for cameras with recordingEnabled=true
       - On module destroy: stop all FFmpeg processes

    3. Create `recording-cleanup.service.ts`:
       - @Cron('0 3 * * *') — runs daily at 3 AM
       - `cleanup()` — reads all org recording configs, for each camera deletes segments older than retentionDays. Uses `fs.readdir` on `/mnt/recordings/{cameraId}/` to find date folders older than retentionDays and removes them. Logs cleanup stats.
       - `getStorageUsage(orgId)` — returns { total: number, used: number, percent: number } by scanning recording dirs

    4. Create `recording.controller.ts`:
       - `GET /recording/config` — get recording config for org
       - `PATCH /recording/config` — update recording config (ZodValidationPipe(recordingConfigSchema), ADMIN/SUPER_ADMIN)
       - `GET /recording/status` — per-camera recording status
       - `GET /events/:eventId/export` — trigger 30s clip export for event (returns download URL or initiates FFmpeg extraction of +/- 15s around event timestamp)
       - `GET /recording/storage` — storage usage

    5. Extend `ingestion.service.ts` — call `recordingService.startRecording()` when a camera is created or its RTSP URL changes.

    6. Add RecordingModule to AppModule imports.

    Handle edge cases: camera offline (don't restart recording), disk full (log error, stop recording for affected cameras), FFmpeg crash (auto-restart with backoff).
  </action>

    ### ShareModule:
    1. Create `share.module.ts` — import JwtModule (for token signing), standard @Module
    2. Create `share.service.ts`:
       - `generateToken(orgId, cameraIds: string[], durationHours: number, createdByUserId: string)` → `{ token, expiresAt }`
         - Creates StreamShare record in DB
         - Signs JWT with payload `{ cameraIds, shareId, type: 'stream_share' }`, expiry = now + durationHours
         - Returns the JWT string and expiry timestamp
       - `verifyToken(token: string)` → `{ cameraIds: string[], shareId: string }` — verifies JWT signature and expiry. Also checks DB status !== 'REVOKED'. Throw UnauthorizedException if invalid/expired/revoked.
       - `revokeShare(shareId)` — sets status to 'REVOKED' in DB
       - `getActiveShares(orgId)` — list non-expired, non-revoked shares
       - `recordAccess(shareId)` — increment accessCount, update lastAccessedAt. Trigger notification to share owner via Socket.IO.
       - @Cron every 15 minutes: mark StreamShare records where expiresAt < now as status='EXPIRED'
    3. Create `share.controller.ts`:
       - `POST /shares` — generate share link (ADMIN/SUPER_ADMIN, ZodValidationPipe(createStreamShareSchema))
       - `GET /shares` — list active shares
       - `DELETE /shares/:id` — revoke share
       - `GET /shares/:id/access-log` — view access count and last access time
       - `GET /stream/share/:token` — NO AUTH required. Verifies token, returns stream URLs for the shared cameras. This is the public no-auth endpoint consumed by share link recipients.

       - `getEventCountByDate(orgId, dateFrom, dateTo)` — returns daily event counts for timeline chart
       - `getEventDetail(eventId)` — full event with video clip reference if available
    3. Create `event.controller.ts`:
       - `GET /events` — search events with query params: dateFrom, dateTo, cameraId, eventType, severity, page, limit
       - `GET /events/stats` — event count by date
       - `GET /events/:id` — event detail

    Add both modules to AppModule imports. Follow existing patterns: paginated responses (`{ data, total, page, limit }`), Zod validation, @Roles guards, French error messages.

## Trust Boundaries
| Boundary | Description |
|----------|-------------|
| Public endpoint → Stream share | Unauthenticated users access /stream/share/:token |
| API → Filesystem | FFmpeg writes HLS segments to mounted volume |

## STRIDE Threat Register
| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-11 | Spoofing | Stream share token | mitigate | JWT-signed tokens with HMAC; DB status check for REVOKED; expiry enforced both at JWT and DB level |
| T-02-12 | Tampering | FFmpeg subprocess args | mitigate | RTSP URL validated at creation time (must be valid rtsp:// URL); cameraId uses UUID format; no shell injection (spawn with args array, not shell string) |
| T-02-13 | Info Disclosure | Recording files on disk | accept | Stored on local client disk, access controlled by OS permissions. VISION is single-tenant at client site |
| T-02-14 | Elevation of Privilege | Multi-user role assignment | mitigate | Secondary users limited to ADMIN or VIEWER roles only; only existing ADMIN/SUPER_ADMIN can create invites |
</threat_model>
