<objective>
Implement per-organization hash-chain audit integrity (FND-06). Each new audit entry is cryptographically linked to the previous entry in the same organization's chain via SHA256. An auditor can call `verifyOrganizationChain(organizationId)` to walk the chain and detect any tampering. This extends the existing entity-level audit system to organization-level hash chains per D-11.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AuditLog write → Hash chain | Hash computation is deterministic; any modification to an entry breaks the chain |
| AuditLog read → verifyOrganizationChain | Verification reads raw SQL; RLS may restrict access for non-admin users |
| Client → verifyOrganizationChain API | Auditor must have organization membership to verify; endpoint scoped by RolesGuard |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-30 | Tampering | AuditLog entry modification | mitigate | SHA256 hash chain detects any modification. Changing one entry invalidates the hash of that entry AND all subsequent entries. |
| T-04-31 | Repudiation | Orphan audit entries (no chain) | mitigate | Every new audit entry MUST have a valid hash. Genesis entries use "genesis" as previousHash. If hash is missing, chain verification catches it. |
| T-04-32 | Tampering | Hash chain truncation (delete last N entries) | accept | Truncation is detectable if auditor knows expected latestHash. No built-in mechanism for storing expected latestHash externally — this is acceptable per D-11 (standalone chain verification). |
| T-04-33 | Information Disclosure | verifyOrganizationChain — cross-org access | mitigate | $queryRawUnsafe with explicit WHERE organizationId = $1. RLS also enforces isolation. Auditor must be member of the organization. |
| T-04-34 | Denial of Service | Large audit chain verification (100K+ entries) | mitigate | Chain verification is O(n). For large orgs, verification walks all entries. Consider background scheduling (deferred). Not a concern for MVP — Phase 4 orgs will have small audit logs. |
</threat_model>
