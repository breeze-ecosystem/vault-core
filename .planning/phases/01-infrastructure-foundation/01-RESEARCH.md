# Phase 1: Infrastructure Foundation — Research

**Researched:** 2026-07-17
**Domain:** Python async I/O, MQTT infrastructure, Docker hardware networking
**Confidence:** HIGH

## Summary

Phase 1 delivers the foundational infrastructure for all hardware integration: an async Edge Agent rewrite (INF-01), Mosquitto MQTT security hardening (INF-02), and Docker networking for serial/multicast passthrough (INF-03). The current Edge Agent (`edge/agent/agent.py`) is synchronous, using blocking `httpx` calls with a `schedule`-based polling loop. It must be rewritten to use `asyncio` with `aiomqtt` for MQTT, `pyserial-asyncio` for OSDP serial I/O, and `aiohttp` for HTTP calls — all in a single event loop. The Mosquitto service exists in `docker-compose.yml` but has zero security configuration (no TLS, no authentication, no ACLs). Docker networking requires `--network=host` for the Edge Agent to access serial devices and multicast UDP for ONVIF WS-Discovery.

**Critical finding:** The recommended `asyncio-mqtt` package is **archived** (last release Jun 2023). The same maintainer has continued development as **`aiomqtt`** v2.5.1 (Mar 2026), which has an identical API but is actively maintained. The plan must use `aiomqtt>=2.5.1` instead.

**Primary recommendation:** Use `aiomqtt>=2.5.1`, `pyserial-asyncio>=0.6`, and `aiohttp>=3.14.1`. Implement asyncio-based Edge Agent with graceful lifecycle management via SIGTERM/SIGINT handlers. Secure Mosquitto with TLS + username/password auth using a generated self-signed CA. Use `--network=host` + `--device` flags for Docker hardware access.

### Locked Decisions
- D-01: Use native Python `asyncio` as the async runtime foundation with `aiohttp` for HTTP, `asyncio-mqtt` for MQTT, and `pyserial-asyncio` for serial I/O — single event loop, single process, all coroutines.

- D-03: Implement graceful lifecycle management: SIGTERM/SIGINT triggers orderly shutdown — drain MQTT, close serial ports, flush pending HTTP requests, then exit.
- D-04: Docker image stays `python:3.12-slim` (keep existing pattern).

- D-10: Edge Agent uses `--network=host` for direct serial device access and multicast support. Trade-off: no container network isolation accepted for edge deployment.

### Pattern 1: Async Task Lifecycle
**What:** Each protocol (serial, MQTT, HTTP) runs as a separate `asyncio.Task` with structured lifecycle via `asyncio.Event` shutdown signals.

```python
# Source: [ASSUMED — standard asyncio pattern verified via official Python docs]
async def serial_reader(shutdown: asyncio.Event, port: str) -> None:
    """Read OSDP frames from serial port."""
    reader, writer = await serial_asyncio.open_serial_connection(url=port, baudrate=9600)
    try:
        while not shutdown.is_set():
            line = await reader.readline()
            # process OSDP frame...
    finally:
        writer.close()
        await writer.wait_closed()

async def main() -> None:
    shutdown = asyncio.Event()
    loop = asyncio.get_event_loop()
    for sig in (signal.SIGTERM, signal.SIGINT):
        loop.add_signal_handler(sig, shutdown.set)
    
    tasks = [
        asyncio.create_task(serial_reader(shutdown, "/dev/ttyUSB0")),
        asyncio.create_task(mqtt_handler(shutdown)),
        asyncio.create_task(http_reporting(shutdown)),
    ]
    await asyncio.gather(*tasks, return_exceptions=True)
```

### Pattern 2: aiomqtt Reconnect Loop
**What:** Wrap the `aiomqtt.Client` context manager in a retry loop for automatic reconnection. Buffer messages during disconnection and replay on reconnect.

```python
# Source: [VERIFIED: aiomqtt PyPI README — documented reconnection pattern]
import asyncio
import aiomqtt

async def mqtt_handler(shutdown: asyncio.Event, broker: str, port: int, 
                       username: str, password: str, tls_params: aiomqtt.TLSParameters) -> None:
    buffer: list[bytes] = []
    reconnect_interval = 5
    
    while not shutdown.is_set():
        try:
            async with aiomqtt.Client(
                hostname=broker,
                port=port,
                username=username,
                password=password,
                tls_params=tls_params,
            ) as client:
                # Replay buffered messages
                while buffer and not shutdown.is_set():
                    await client.publish("site/...", payload=buffer.pop(0))
                # Subscribe and listen
                async with client.messages() as messages:
                    await client.subscribe("site/{siteId}/door/+/cmd")
                    async for message in messages:
                        if shutdown.is_set():
                            break
                        # route to door controller via serial task...
        except aiomqtt.MqttError as e:
            if shutdown.is_set():
                break
            log.warning("MQTT disconnected: %s — reconnecting in %ds", e, reconnect_interval)
            await asyncio.sleep(reconnect_interval)
```

### Anti-Patterns to Avoid
- **Blocking the event loop with `time.sleep()`:** Use `await asyncio.sleep()` everywhere inside async functions.
- **Mixing sync `httpx` in async tasks:** Use `aiohttp.ClientSession()` for all HTTP calls from the Edge Agent.
- **Using `asyncio-mqtt` instead of `aiomqtt`:** The `asyncio-mqtt` package is archived. Use `aiomqtt` — same API, same maintainer, actively updated.
- **Missing `try/finally` in serial port handling:** Serial ports MUST be properly closed on shutdown to avoid hanging the kernel driver.
- **Plaintext MQTT on WAN:** Never expose port 1883 to non-local networks. Only TLS port 8883 should be accessible from edge agents.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Async MQTT client | Custom asyncio MQTT wrapper around paho-mqtt | `aiomqtt` | Handles reconnection, message filtering, graceful disconnect, TLS params, async generator for messages |
| Async serial I/O | Custom asyncio serial wrapper | `pyserial-asyncio` | Provides `serial_asyncio.open_serial_connection()` returning `StreamReader`/`StreamWriter` — exactly matches asyncio patterns |
| Async HTTP client | Custom socket-based HTTP | `aiohttp` | Connection pooling, timeouts, redirect handling, streaming uploads — all the edge cases `requests` handles but async |
| MQTT message de-duplication | Custom sequence logic | Keep existing in `MqttService` | API already implements sequence number dedup in `mqtt.service.ts` lines 82-93. Edge Agent only needs to include `sequence` field. |
| TLS certificate generation | Custom openssl wrapper | Shell script + `openssl` in Docker entrypoint | One-time setup: generate CA, server cert, client certs at deployment init |
| OSDP protocol library | Custom OSDP implementation | Use existing open-source `pyosdp` or `libosdp` | OSDP is a complex stateful protocol. Don't build it from scratch — Phase 2 will integrate it. Phase 1 lays the serial async infrastructure. |

**Key insight:** The async infrastructure is standard Python ecosystem work. The complex domain logic (OSDP protocol parsing, ONVIF SOAP/WS-Discovery) comes in Phase 2. This phase is purely about getting the networking, lifecycle, and I/O plumbing right.

## Common Pitfalls

### Pitfall 1: Using Archived `asyncio-mqtt`
**What goes wrong:** `pip install asyncio-mqtt` installs an archived package (last release 2023). It works but has no future updates, no security patches.
**Why it happens:** The decision (D-01) names `asyncio-mqtt`. The package was renamed to `aiomqtt` by the same maintainer.
**How to avoid:** Use `aiomqtt>=2.5.1` in requirements.txt. The API is identical — just change the import name from `asyncio_mqtt` to `aiomqtt`.
**Warning signs:** If a package on PyPI shows "This project has been archived," check for a rename.

### Pitfall 2: Blocking Event Loop with Sync I/O
**What goes wrong:** One slow serial read blocks MQTT message processing or health reporting. The whole agent becomes unresponsive.
**Why it happens:** asyncio is single-threaded. Any blocking call (sync `time.sleep()`, sync `requests.get()`, sync `serial.read()`) freezes all tasks.
**How to avoid:** Every I/O call must use the async variant — `await asyncio.sleep()`, `aiohttp` for HTTP, `pyserial-asyncio` for serial, `aiomqtt` for MQTT. Set explicit timeouts on all operations.
**Warning signs:** Tasks that fail to yield control, watchdog timeouts on edge agent health checks.

### Pitfall 3: Mosquitto TLS Certificate Management
**What goes wrong:** Self-signed certs expire. Docker secrets aren't refreshed. Clients fail to connect after cert rotation.
**Why it happens:** TLS certs generated once at setup are forgotten until they break.
**How to avoid:** Generate certs with 10-year validity initially. Store the CA private key securely (Docker secret or vault). Document the renewal command. Use a consistent file path structure: `/mosquitto/certs/{ca,server}.{crt,key}`.
**Warning signs:** MQTT connection errors like `TLS error: certificate expired`, `TLS error: unknown CA`.

### Pitfall 4: Host Network Mode Port Conflicts
**What goes wrong:** `--network=host` exposes ALL container ports directly on the host. A service inside the container listening on port 8883 clashes with host services.
**Why it happens:** Host mode removes Docker's port isolation.
**How to avoid:** The Edge Agent itself doesn't listen on ports (it's a client-only). Verify no services inside the container bind to ports. Use `--network=host` only for Edge Agent, not for other services.
**Warning signs:** Port already in use errors on container start.

### Pitfall 5: Deadly Embrace Between MQTT and Serial
**What goes wrong:** Serial read blocks waiting for data, preventing MQTT from processing incoming door commands. Door commands time out.
**Why it happens:** Both tasks share the same event loop. A long serial read with no timeout starves the MQTT handler.
**How to avoid:** Set `asyncio.wait_for()` timeouts on all serial reads. The serial task should yield to the event loop periodically. Use `as_completed` or `wait(..., FIRST_COMPLETED)` for time-sensitive operations.
**Warning signs:** Door commands sent via MQTT take >1s to reach the controller.

## Common Pitfalls (additional)

### Pitfall 6: Graceful Shutdown of asyncio Tasks
**What goes wrong:** Container receives SIGTERM but the Python process doesn't propagate it to all asyncio tasks. Serial ports stay open, MQTT broker sees unclean disconnect, pending HTTP requests are dropped.
**Why it happens:** `asyncio.run()` catches SIGINT but Docker sends SIGTERM. Without explicit signal handlers, tasks just get killed.
**How to avoid:** Register `loop.add_signal_handler(signal.SIGTERM, ...)` to set a shutdown event. All async tasks must check this event in their loop condition. Use `asyncio.gather(*tasks)` with the shutdown event.
**Warning signs:** Mosquitto logs show "Client X disconnected unexpectedly." Serial port `/dev/ttyUSB0` remains locked after container stops.

### Pitfall 7: In-memory Event Buffer Overflow
**What goes wrong:** Prolonged MQTT disconnection causes unbounded memory growth from buffered events.
**Why it happens:** D-16 specifies an in-memory bounded queue, but without explicit bounds enforcement, the buffer grows without limit.
**How to avoid:** Use `asyncio.Queue(maxsize=5000)` (or similar). When the queue is full, log a warning and drop the oldest event (FIFO eviction) instead of crashing with memory exhaustion.
**Warning signs:** Agent memory usage grows linearly during WAN outages.

## Open Questions

1. **Should we keep `httpx` alongside `aiohttp` for backward compatibility?**
   - What we know: Current `agent.py` uses sync `httpx` for heartbeat/health. Some health check operations may still suit sync.
   - What's unclear: Whether all HTTP operations should migrate to `aiohttp` or if some SHOULD remain sync (e.g., Docker SDK calls).
   - Recommendation: Replace all sync `httpx` with `aiohttp` in the rewrite. Remove `httpx` and `schedule` from requirements.txt. Docker SDK (`docker>=7.0`) is already async-compatible.

2. **Where should TLS certificates be generated — Docker entrypoint or compose init container?**
   - What we know: Self-signed CA + certs are needed. Docker secrets mechanism exists for secure distribution.
   - What's unclear: Whether to use an init container pattern (generate certs in a `setup` container that runs before Mosquitto) or a script in the Mosquitto image entrypoint.
   - Recommendation: Use a separate `mosquitto-init` service in docker-compose that generates certs and writes them to a shared volume, then exits. Mosquitto depends on this service completing.

3. **How are per-agent credentials (D-15) generated and distributed?**
   - What we know: D-15 says per-agent credentials generated by API during enrollment. D-09 says rotation via env var update + Docker restart.
   - What's unclear: Whether this enrollment API endpoint exists or needs to be built in Phase 1.
   - Recommendation: For Phase 1, use static credentials set via env vars (matching current pattern). The API enrollment endpoint is a Phase 2 task. Document the static approach as temporary.

### Known Threat Patterns for MQTT + Docker

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| MQTT broker exposed without auth | Tampering, Info Disclosure | `allow_anonymous false`; `password_file` with hashed credentials; listener bound to specific interface |
| Man-in-the-middle on MQTT | Spoofing, Tampering, Info Disclosure | TLS with `CERT_REQUIRED` on client side; verify server cert against CA |
| Container breakout via serial device | Elevation of Privilege | Use `--device` flags (not `--privileged`); each serial port explicitly mapped |
| Event injection via spoofed MQTT | Spoofing | Per-site ACLs restrict each agent to its own topic namespace; sequence number dedup in API |
| Credential leak in env vars | Information Disclosure | Use Docker secrets for MQTT password; log scrubbing in agent (never log credential values) |
