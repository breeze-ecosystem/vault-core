<decisions>
## Implementation Decisions

### Edge Agent Async Architecture
- **D-01:** Use native Python `asyncio` as the async runtime foundation with `aiohttp` for HTTP, `asyncio-mqtt` for MQTT, and `pyserial-asyncio` for serial I/O — single event loop, single process, all coroutines.
- **D-02:** Edge Agent runs as a single Python process with separate asyncio tasks for serial I/O (OSDP), MQTT pub/sub, HTTP calls to API, and health reporting.
- **D-03:** Implement graceful lifecycle management: SIGTERM/SIGINT triggers orderly shutdown — drain MQTT, close serial ports, flush pending HTTP requests, then exit.
- **D-04:** Docker image stays `python:3.12-slim` (keep existing pattern).
- **D-05:** Testing uses hybrid approach — unit tests with mocked I/O for business logic, integration tests with an OSDP simulator tool and Mosquitto in Docker Compose.

### MQTT Security
- **D-06:** Authentication via username/password + TLS encryption. No mutual TLS (X.509 client certs) — keeps ops simple.
- **D-07:** TLS uses a self-signed internal CA generated during Docker Compose setup. Certs distributed as Docker secrets or mounted files. No public CA / domain needed.
- **D-08:** Broker-level topic ACLs per site (`site/{siteId}/#`) — each Edge Agent scoped to its site's topic namespace.
- **D-09:** Credential rotation via env var update + Docker restart (simplest, fits Coolify deployment pattern).

### Docker Networking
- **D-10:** Edge Agent uses `--network=host` for direct serial device access and multicast support. Trade-off: no container network isolation accepted for edge deployment.
- **D-11:** Serial devices mapped via explicit `--device` flags per port (e.g., `--device /dev/ttyUSB0:/dev/ttyUSB0`). No privileged mode.
- **D-12:** Edge Agent runs ONVIF WS-Discovery probes and reports discovered cameras to the API via HTTP POST. API does not need direct multicast access.
- **D-13:** Mosquitto runs as a centralized server-side service in Docker Compose. Edge Agents connect to it over the WAN.

### Edge Agent ↔ Mosquitto Topology
- **D-14:** Topic naming uses per-site hierarchical convention: `site/{siteId}/door/{doorId}/state`, `site/{siteId}/door/{doorId}/cmd`, `site/{siteId}/reader/{readerId}/badge`. Matches existing `MqttService` pattern in the API.
- **D-15:** Each Edge Agent authenticates with per-agent credentials generated by the API during enrollment. Mosquitto ACL scopes each agent to its site topics.
- **D-16:** On MQTT disconnection, Edge Agent buffers events in-memory (bounded queue) and replays on reconnect. No persistent queue — brief WAN interruptions tolerated without event loss.

### Agent's Discretion (user deferred)
- D-05 (testing approach), D-08 (topic ACLs), D-09 (rotation strategy), D-11 (serial mapping), D-12 (ONVIF discovery approach), D-15 (per-agent credentials) were recommended by the agent and accepted by default — these are negotiable during planning if technical constraints arise.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **`apps/api/src/mqtt/mqtt.service.ts`**: Existing MQTT client with sequence number dedup, EventEmitter2 routing to domain modules. Topic convention already established — Phase 1's Edge Agent must publish to topics that this service can consume.
- **`edge/agent/agent.py`**: Existing agent structure (health reporting, heartbeat, Docker monitoring) — rewrite must preserve these capabilities while adding async OSDP serial + MQTT.
- **`docker-compose*.yml`**: Mosquitto service already defined but without TLS/auth config. Phase 1 adds security layer.

### Established Patterns
- **Edge Agent as reporting agent**: Current agent collects metrics and POSTs to API. Rewrite adds MQTT pub/sub but keeps the same API reporting channel for health/Docker monitoring.
- **EventEmitter2 routing**: MQTT events → EventEmitter2 → domain modules. Edge Agent's outgoing MQTT topics must align with what MqttService subscribes to.
- **Docker Compose env vars for configuration**: All Edge Agent config (MQTT credentials, API URL, site ID) via env vars.

### Integration Points
- **Edge Agent → Mosquitto**: MQTT TLS connection (port 8883), per-agent credentials, topic namespace `site/{siteId}/#`
- **Mosquitto → API MqttService**: Existing EventEmitter2 routing unchanged. API already consumes MQTT events via MqttService.
- **Edge Agent → API (HTTP)**: Health reports, heartbeat, ONVIF discovery results POST to `/api/supervision/report` and `/api/heartbeat`
- **Edge Agent → Serial**: RS-485/RS-232 for OSDP door controllers via host `/dev/tty*` devices

</code_context>
