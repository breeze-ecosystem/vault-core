# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2: Hardware Integration. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Key design principle (D-14):** All hardware controls integrate into existing Dashboard views — door controls in the Doors page, PTZ in the Camera viewer, controller enrollment in the Equipment section. No separate "Hardware" silo.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn/ui (default style, slate base color, CSS variables) |
| Preset | `style: default, baseColor: slate, cssVariables: true, rsc: true` |
| Component library | Radix UI primitives (via shadcn/ui) |

**Animation library:** `motion` (motion/react) — already used in cameras page for transitions

## Spacing Scale

All values follow Tailwind defaults (4px base). Verified against `tailwind.config.ts` and existing component code.

| Token | Tailwind | Value | Usage |
|-------|----------|-------|-------|
| xs | `gap-1` / `p-1` | 4px | Icon-to-text gaps, inline badge padding |
| sm | `gap-2` / `p-2` | 8px | Compact element spacing between buttons, filter pill padding |
| md | `gap-3` / `p-3` | 12px | Default element spacing in door cards, PTZ button grid |
| base | `gap-4` / `p-4` / `space-y-4` | 16px | Card content padding, section spacing |
| lg | `gap-6` / `p-6` / `space-y-6` | 24px | Page section padding (existing `space-y-6` on Doors page) |
| xl | `gap-8` | 32px | Layout gaps between major sections |
| 2xl | `p-8` | 48px | Modal padding, large empty states |

Exceptions:
- **PTZ directional pad buttons:** 44px × 44px minimum touch targets per WCAG 2.5.8 (overrides 40px button default)
- **Event journal inline thumbnail:** 64px × 48px (wider aspect ratio for camera stills)

## Component Inventory: What to Modify

### 1. Door Card (in `portes/page.tsx`)
**Current state:** Card with state icon + label + zone name + controller ID + "Alerte" button. No direct lock/unlock.

**Modifications (D-04, D-05, D-11):**

| Element | Change | States | Interaction |
|---------|--------|--------|-------------|
| Lock/Unlock buttons | Add inline button pair inside `CardContent` below state line | Locked → shows "Déverrouiller" button; Unlocked/Held-open/Forced → shows "Verrouiller" button; Desynchronized → both disabled | One-click toggle via `POST /api/doors/:id/cmd` with body `{ command: "lock" \| "unlock" }` |
| Command state indicator | Add 3-dot state label next to button | `hidden` (idle) → `Envoi...` (sending) → `Envoyé` (sent) → `Acquitté` (acknowledged) → `hidden` or `Échec` (on error with retry prompt) | Auto-advance via Socket.IO event; auto-retry once on no ack within 2s |
| Zone dropdown | Replace static `{door.zoneName}` with inline `<select>` | Same zone list as page-level filter | On change: `PATCH /api/doors/:id/zone` with `{ zoneId }`; toast confirmation |
| Unauthorized state | Hide lock/unlock buttons if user role is `VIEWER` or `AUDITOR` | Buttons absent | Role check from `useAuth()` |

**Button variant rules:**
- Lock: `variant="outline" size="sm"` with `Lock` icon. On unlocked/held-open/forced doors: `className="border-success/30 text-success hover:bg-success/10"`
- Unlock: `variant="outline" size="sm"` with `Unlock` icon. On locked doors only: `className="border-destructive/30 text-destructive hover:bg-destructive/10"`
- Disabled state (desynchronized): `disabled` + tooltip "Porte désynchronisée — action impossible"

### 2. Bulk Operations Bar (in `portes/page.tsx`, existing `bulk-action-bar.tsx`)
**Current state:** Alert-selection bulk action bar (acknowledge/resolve).

**Modifications (D-12):**

| Element | Change | States | Interaction |
|---------|--------|--------|-------------|
| Bulk Lock/Unlock controls | Add zone-level buttons — "Verrouiller la zone" / "Déverrouiller la zone" per zone | Visible when zone filter is active; hidden when "Toutes les zones" selected | Button click opens `confirmation-dialog.tsx` with: title `"Verrouillage de zone"`, description `"Verrouiller toutes les portes de {zoneName} ? Cette action est irréversible."` |
| Bulk Lock confirm | Red variant button | `"Verrouiller toutes les portes"` | Calls `POST /api/doors/zones/:id/lockdown` |
| Bulk Unlock confirm | Warning variant button | `"Déverrouiller toutes les portes"` | Calls `POST /api/doors/zones/:id/emergency-unlock` |
| Success | Toast + door cards animate | "Zone verrouillée" / "Zone déverrouillée" | Socket.IO `emergency-update` updates all door cards |
| Error | Toast with error + door cards revert | "Échec — {reason}" | Cards revert to previous state |

**Confirmation dialog pattern (from existing `confirmation-dialog.tsx`):**
- Add `variant` prop to support `"destructive"` (red) for lock, `"warning"` (yellow) for unlock
- Required role: `ADMIN` or `SUPER_ADMIN` (block if below)
- On confirm: lock button text → `"Verrouillage..."` with spinner, unlock → `"Déverrouillage..."` with spinner

### 3. Camera Viewer PTZ Overlay (in `components/video-player.tsx`)
**Current state:** Video player with WebRTC/HLS streams, fullscreen toggle, connection state display.

**Modifications (D-06, D-07, D-16):**

| Element | Change | States | Interaction |
|---------|--------|--------|-------------|
| PTZ overlay container | Add absolute-positioned overlay on top of video | `opacity-0` (hidden, after 4s inactivity) → `opacity-100` (visible, on mouse enter or touch) | `onMouseEnter` / `onTouchStart` show; `onMouseLeave` starts 4s timeout to hide |
| Directional pad | 4-direction cross + center stop button, 44×44px touch targets | Inactive (default) → Active direction highlighted in `primary` (on hover/touch) | `POST /api/cameras/:id/ptz/continuous { pan, tilt, zoom: 0 }` — STOP on center button or release |
| Zoom slider | Vertical slider on right edge of player | Thumb at 0 (default) → dragged up (zoom in) / down (zoom out) | `POST /api/cameras/:id/ptz/continuous { pan: 0, tilt: 0, zoom: value }` — rate-limited client-side to 5 cmd/s |
| Preset thumbnail bar | Collapsible horizontal bar below video | Collapsed (default, 32px strip showing "N présents") → Expanded (thumbnails visible) | Toggle via chevron button; each thumbnail is 64×48px with name label |
| Preset thumbnail item | Image + name + recall | Default (dim) → Hover (brightness increase) → Click recalls preset | `POST /api/cameras/:id/ptz/goto-preset { presetToken }` with loading state |
| Save preset button | "+" icon button at end of thumbnail bar | Normal → disabled (max 10 presets) | Opens inline name input → `POST /api/cameras/:id/ptz/save-preset { name }` |
| Role gate | Entire overlay hidden for non-supervisors | `null` (role < SUPERVISOR) → PTZ controls rendered (SUPERVISOR+) | Check `user.role` — must be `ADMIN`, `SUPER_ADMIN`, or `SUPERVISOR` |
| No-PTZ camera | Overlay absent with tooltip | Camera has `ptzCapabilities.hasPtz === false` → no overlay rendered | Tooltip on camera card: "Cette caméra ne supporte pas PTZ" |

## New Components Required

### `ptz-controls.tsx`
Props interface:
```typescript
interface PTZControlsProps {
  cameraId: string;
  hasPtz: boolean;
  presets?: PTZPreset[];
  userRole: string;
  onMove: (pan: number, tilt: number, zoom: number) => void;
  onStop: () => void;
  onGotoPreset: (presetToken: string) => void;
  onSavePreset: (name: string) => void;
  disabled?: boolean;
}
```

States:
- **Hidden** (role < SUPERVISOR or `hasPtz === false`)
- **Inactive** (visible, no movement, dim)
- **Active** (direction highlighted, movement in progress)
- **Loading** (sending command, brief spinner on direction button)
- **Disabled** (`disabled` prop, e.g., camera offline)
- **Auto-hiding** (fading out after 4s inactivity)

### `door-card.tsx` (extracted from `portes/page.tsx`)
Current door card logic is inline in the page. Extract to standalone component.

Props interface:
```typescript
interface DoorCardProps {
  door: DoorStateDto;
  zones: ZoneDto[];
  userRole: string;
  onLock: (doorId: string) => void;
  onUnlock: (doorId: string) => void;
  onZoneChange: (doorId: string, zoneId: string) => void;
  onAlertConfig: (door: DoorStateDto) => void;
  isAdmin: boolean;
  heldOpenSeconds: number | null;
  commandState?: CommandState;  // 'idle' | 'sending' | 'sent' | 'acknowledged' | 'failed'
}
```

States:
- **Idle** — normal display, buttons inactive until hover
- **Sending** — button loading spinner, label "Envoi..."
- **Sent** — success icon animation, label "Envoyé"
- **Acknowledged** — green checkmark, label "Acquitté"
- **Failed** — red X, "Échec — Réessayer" button
- **Desynchronized** — both action buttons disabled with tooltip
- **Pulse** (forced/held-open alarm state) — existing `animate-pulse` behavior

## Error & Edge Case States

| Scenario | UI State | Copy | Recovery |
|----------|----------|------|----------|
| Door command fails (no ack) | Command indicator → "Échec" + "Réessayer" | "Échec — {reason}" | Click retry; auto-retry already happened once |
| Camera has no PTZ | Overlay not rendered, tooltip on camera | "Cette caméra ne supporte pas PTZ" | None (capability is fixed) |
| User below SUPERVISOR views camera | No PTZ overlay, tooltip | "Contrôle PTZ réservé aux superviseurs" | None (role-gated at API + UI) |
| PTZ rate limit hit | Press direction button → no visual feedback (command queued) | Client-side throttle prevents UI feedback loss | Next input accepted after 200ms cooldown |
| Controller goes offline | Status → OFFLINE (red badge) | "Hors ligne" | Auto-update via health topic |
| Controller enrollment API fails | Modal error state | "{error}" + "Fermer" | User can retry |
| Zone dropdown save fails | Toast error, revert dropdown to previous value | "Échec de la mise à jour de zone — {reason}" | Retry by selecting again |
| Preset save fails | "+" button re-enables, toast error | "Échec de l'enregistrement du preset" | Retry |
| Camera snapshot unavailable for thumbnail | Placeholder icon (64×48 grey rect with `Video` icon) | — | No thumbnail shown |
