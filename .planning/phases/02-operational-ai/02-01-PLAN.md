Deliver the Incident Management core vertical slice — end-to-end from database models through API to dashboard UI. By the end of this plan, operators can create, triage, assign, comment on, and close incidents following a strict lifecycle state machine. The system auto-triages HIGH/CRITICAL alerts into incidents via the event bus and manages SLA escalation timers through BullMQ delayed jobs.

Purpose: Incident management is the operational backbone of the security platform. Every alert, event, and observation flows into incidents for structured response. This plan delivers the full lifecycle with proper state enforcement, assignment workflow, and real-time collaboration.

Output: Working incident CRUD + lifecycle management with dashboard UI, auto-triage from alerts, assignment with SLA timers, and real-time status updates via Socket.IO.

    Create service with the following methods. Follow existing NestJS service patterns (inject PrismaService, EventEmitter2, ConfigService, @InjectQueue):

    1. `create(dto, user)`: Creates incident with Prisma, generates `incident.state-changed` event bus event, returns incident with relations. Log to incident_events hypertable via $queryRaw on creation.

       - Call `IncidentStateMachine.validateTransition(currentStatus, newStatus)`
       - If invalid, throw `BadRequestException("Invalid transition from ${current} to ${proposed}")`
       - Update Prisma Incident record
       - Insert row into incident_events hypertable via $queryRaw
       - If transitioning to CLOSED, set `closedAt = new Date()`
       - If transitioning from RESOLVED back to INVESTIGATING, clear `closedAt`
       - Emit `incident.status-changed` event
       - Return updated incident

    5. `assignIncident(id, userId, assignedBy)`: Assign incident to a user, create IncidentAssignment record, update incident status to TRIAGE if currently OPEN, set assignedAt, emit `incident.assigned` event.     6. `addComment(id, text, user)`: Create IncidentComment with Prisma, emit `incident.comment-added` event, return comment with user info.

    7. `getComments(id)`: Fetch all comments for an incident, ordered by createdAt ascending, with user name included.

    8. `getAssignmentHistory(id)`: Fetch all assignments for an incident, ordered by assignedAt descending, with user details.

    9. `getStatusHistory(id)`: Query incident_events hypertable for all status transitions of this incident, ordered by time ascending.

    10. `onModuleInit()`: On server startup, re-evaluate all open incidents' SLA timers. Query incidents where status not in (RESOLVED, CLOSED) and assignedAt is not null.
