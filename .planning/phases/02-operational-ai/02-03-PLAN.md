<objective>
Deliver Visitor Management as a complete vertical slice. By the end of this plan, hosts can pre-register visitors with visit details and zone restrictions, visitors receive QR credentials valid for their visit duration, security can process check-in/check-out at reception, and all visitor activity is logged and correlated with video through the existing event pipeline.

Purpose: Visitor management extends Phase 1's credential system with time-limited visitor access. It completes the access control picture — the platform can now manage not just employees with permanent credentials but also temporary visitors with scoped access.

Output: Full visitor lifecycle from pre-registration through check-out with QR credential generation, zone restriction enforcement, video correlation, and dashboard management pages.
</objective>

    **Part A — Visitor Service (visitor.service.ts):**

    Create service following RESEARCH.md Pattern 4 exactly. Inject PrismaService, EventEmitter2, and AccessService (from Phase 1 modules/access/access.service.ts).

    1. `preregister(dto, user)`: RESEARCH.md lines 600-653 EXACT implementation:
       - Create Visitor record via Prisma
       - Create QR-type Credential via `accessService.createCredential()` with:
         - userId: dto.hostUserId (credential linked to host)
         - type: "QR", qrSeed: crypto.randomUUID()
         - validFrom: dto.validFrom, validUntil: dto.validUntil
         - maxUses: null (unlimited during visit)
       - If dto.zoneIds has entries, call `accessService.createAccessLevel()` for each zone with a default 24/7 schedule (fetch or create a schedule named "24/7" for the site if it doesn't exist)
       - Generate QR badge via `accessService.generateQrCode(credential.id)`
       - Create Visit record with status="scheduled"
       - Emit `visitor.preregistered` event via EventEmitter
       - Return { visit, qrCode }

    2. `checkIn(visitId, user)`:
       - Fetch visit with visitor + credential relations
       - Validate visit status is "scheduled" (throw BadRequestException if not)
       - Validate validFrom <= now <= validUntil (throw if outside visit window)
       - Update credential: set isActive=true (Phase 1 AccessService already checks this)
       - Update visit: status="active", checkedInAt=new Date()
       - Emit `visitor.checked-in` event with visitId + visitorId + siteId
       - Return updated visit

    3. `checkOut(visitId, user)`:
       - Fetch visit with credential relation
       - Validate visit status is "active" (throw BadRequestException if not)
       - **Critical for Pitfall 3 avoidance**: Immediately deactivate credential: set isActive=false
       - Also add credential ID to Redis revoked set: key `credential:revoked:{credentialId}` with TTL = remaining validUntil duration
       - Update visit: status="completed", checkedOutAt=new Date()
       - Emit `visitor.checked-out` event
       - Return updated visit

    4. `cancelVisit(visitId)`:
       - Cancel a scheduled visit (before check-in)
       - Deactivate credential (isActive=false)
       - Set visit status="cancelled"
       - Emit `visitor.cancelled` event

    5. `listVisits(filters)`:
       - Query Visit with filters: status, hostUserId, from/to date range, page, limit
       - Include visitor and host relations
       - Site-scoped via user's siteId (filter host user's site)
       - Paginated response { data, total, page, limit }

    6. `getVisit(id)`:
       - Single visit with all relations: visitor, host, credential

    7. `getVisitor(id)`:
       - Single visitor with visit history

    8. `listVisitors(params)`:
       - Query Visitor with name/email search

    9. **OnModuleInit — Cleanup stale visitor credentials:**
       - On startup, find all visits where status="active" but checkedOutAt is null AND validUntil < now
       - For each: deactivate credential, set status="completed", set checkedOutAt=validUntil
       - Log the number of auto-cleaned visits
       - This handles the edge case where server restarts with expired active visits
