    Create service following RESEARCH.md Pattern 7 (lines 883-967). Inject PrismaService, EventEmitter2, @InjectQueue('equipment-health').

    1. `@Cron("*/30 * * * * *") checkCameraHealth()` (30-second intervals):
       - Threshold: cameras with no heartbeat in 5 minutes are stale
       - Query Prisma for cameras where isRecording=true AND (lastHeartbeat is null OR lastHeartbeat < NOW() - INTERVAL '5 minutes')
       - For each stale camera:
         - Insert into camera_health hypertable via $queryRaw with status='offline'
         - Update Camera model: status="OFFLINE" (if previously ONLINE/DEGRADED)
         - Emit `equipment.alert` event with deviceType='camera', deviceId, status='offline'

    2. `@Cron("*/30 * * * * *") checkReaderHealth()`:
       - Similar pattern: check reader_health hypertable for stale readers
       - A reader is stale if no health event received in last 10 minutes
       - Insert offline record and emit equipment.alert

    3. **MQTT Health Event Handler:**
       - `@OnEvent("mqtt.controller.health", { async: true })` — handles controller health payloads from MQTT
       - Parse payload: extract batteryLevel, connectionStability, firmwareVersion, cpuLoad, memoryUsage
       - Insert into controller_health hypertable via $queryRaw
       - Alert conditions:
         - batteryLevel < 20% → emit equipment.alert with status='low-battery'
         - connectionStability != 'stable' → emit equipment.alert with status='connection-issue'
       - Apply debounce: Redis key `equipment:debounce:controller:{controllerId}:{alertType}` with 5-min TTL

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MQTT → EquipmentService | Controller/reader health data from MQTT topics |
| API → pgcrypto | Encryption key crosses service boundary |
| Database → pruning cron | Data deletion crosses storage boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-22 | Spoofing | MQTT health topic spoofing | mitigate | MQTT topics use Phase 1 validated topic structure; health payload validated before DB insert |
| T-02-23 | Information Disclosure | Encryption key exposure | mitigate | ENCRYPTION_KEY stored in environment variable, never in database or logs; key rotation requires maintenance window (Pitfall 5) |
| T-02-24 | Denial of Service | Alert fatigue from equipment alerts | mitigate | Redis debounce keys per device+alertType with configurable TTL (Pitfall 6 mitigation: 60s for cameras, 5min for controllers) |
| T-02-25 | Tampering | Retention policy (data deletion) | mitigate | @Roles(ADMIN) on all retention policy mutation endpoints; deletion logged via audit |
| T-02-26 | Information Disclosure | Retention pruning deletes wrong data | mitigate | Event type is validated against known values; DELETE is parameterized via $queryRawUnsafe with cutoff bound |
</threat_model>
