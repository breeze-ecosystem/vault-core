<objective>
Deliver the Recurring Pattern Detection vertical slice — end-to-end from TimescaleDB frequency analysis queries through the pattern detection engine to the patterns dashboard. By the end of this plan, the system surfaces recurring situations (repeated door forced-open events, reader failure clusters, camera FPS degradation) with configurable thresholds, Redis deduplication, and event bus integration.

Purpose: Recurring pattern detection identifies chronic issues that might be missed as individual events. A door forced open once is an alert; the same door forced open 5 times in an hour indicates a systemic problem (misconfiguration, broken hardware). This fills the gap between individual event alerts and strategic risk scoring.

Output: Working pattern detection engine with configurable frequency rules, detected_patterns hypertable, REST API for querying patterns, and a patterns dashboard for reviewing surfaced issues.

    Create BullMQ processor for `recurring-patterns` queue (follows incident.processor.ts pattern):

    ```typescript
    @Processor('recurring-patterns')
    export class PatternsProcessor extends WorkerHost {
      private readonly logger = new Logger(PatternsProcessor.name);

      constructor(private patternsService: PatternsService) {
        super();
      }

      async process(job: Job): Promise<any> {
        switch (job.name) {
          case 'detect-patterns':
            await this.patternsService.detectPatterns();
            return { completed: true };
          case 'resolve-pattern':
            // Job data: { patternId: string, deviceId: string }
            await this.patternsService.resolvePattern(job.data.patternId);
            return { resolved: true };
          default:
            return { skipped: true, reason: `Unknown job name: ${job.name}` };
        }
      }
    }
    ```
