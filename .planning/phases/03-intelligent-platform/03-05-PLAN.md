<objective>
Deliver the Multi-Site Isolation and Maintenance Workflows vertical slice — end-to-end from PostgreSQL RLS policies and Prisma schema changes through the maintenance ticket engine to the unified dashboard. By the end of this plan, multi-site data isolation is enforced at the database level via RLS, equipment degradation auto-creates maintenance tickets routed to the maintenance team, and operators can track maintenance tickets alongside security incidents in a unified view.

Purpose: Multi-site isolation is a compliance requirement (AUDT-06) — without it, operators at one site could see data from other sites. Maintenance workflows close the loop on equipment health: when the predictive engine (Plan 04) or real-time health checks detect degradation, the system automatically creates maintenance tickets and routes them to the right team.

Output: Working RLS policies on all site-scoped tables, SiteContextMiddleware, extended Incident model with MAINTENANCE_TICKET support, maintenance auto-creation on equipment alerts, and a unified maintenance dashboard.
</objective>

    validateTransition(current: string, proposed: string, ticketType?: string): string {
      if (current === proposed) return proposed;

      const transitions = ticketType === 'MAINTENANCE_TICKET'
        ? MAINTENANCE_TRANSITIONS
        : INCIDENT_TRANSITIONS;

      const allowed = transitions[current as IncidentStatus];
      if (!allowed) {
        throw new BadRequestException(`Unknown status: ${current}`);
      }
      if (!allowed.includes(proposed as IncidentStatus)) {
        throw new BadRequestException(
          `Invalid transition from ${current} to ${proposed} for ${ticketType || 'SECURITY_INCIDENT'}. ` +
          `Allowed: [${allowed.join(", ")}]`,
        );
      }
      return proposed;
    }

    ```typescript
    import { Injectable, NestMiddleware, Logger } from "@nestjs/common";
    import { FastifyRequest, FastifyReply } from "fastify";
    import { PrismaService } from "../prisma/prisma.service";

    @Injectable()
    export class SiteContextMiddleware implements NestMiddleware {
      private readonly logger = new Logger(SiteContextMiddleware.name);

      constructor(private prisma: PrismaService) {}

      async use(req: FastifyRequest, _res: FastifyReply, next: () => void) {
        const user = (req as any).user;
        if (user?.siteId) {
          try {
            await this.prisma.$executeRawUnsafe(
              `SELECT set_config('app.current_site_id', $1, TRUE)`,
              user.siteId,
            );
          } catch (err: any) {
            // PostgreSQL session variable setting is non-critical — log and continue
            this.logger.warn("Failed to set RLS context", err.message);
          }
        }
        // If user has no siteId (super-admin), don't set the context
        // RLS policy returns no rows for NULL current_site_id, but super-admin uses
        // a separate mechanism (application-level bypass)
        next();
      }
    }
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API maintenance endpoints | Unauthorized access to maintenance ticket CRUD |
| Client → any site-scoped API endpoint | Cross-site data access prevented by RLS |
| Event bus (equipment.alert → maintenance) | Internal event flow auto-creates tickets |
| PostgreSQL RLS per-request context | Session variable tampering could bypass isolation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-18 | Information Disclosure | All site-scoped API endpoints | mitigate | PostgreSQL RLS enforced at database level — even if API query omits site_id filter, RLS prevents cross-site data leakage (AUDT-06) |
| T-03-19 | Elevation of Privilege | RLS bypass via superuser | accept | SUPER_ADMIN role inherently bypasses RLS per design. Document that superusers see all sites |
| T-03-20 | Tampering | SiteContextMiddleware | mitigate | Middleware sets read-only session variable ('local' scope via third param TRUE). Setting failure is non-critical and logged |
| T-03-21 | Denial of Service | Maintenance ticket auto-creation | mitigate | EquipmentService already has debounce (60s TTL) on equipment.alert events. MaintenanceService checks for existing unresolved tickets before creating duplicates |
| T-03-22 | Elevation of Privilege | MAINTENANCE_TEAM role access | mitigate | @Roles(ADMIN, SUPERVISOR, MAINTENANCE_TEAM) restricted on mutation endpoints. MAINTENANCE_TEAM can only transition ticket status and view, not create or assign |
| T-03-23 | Information Disclosure | MaintenanceController endpoints | mitigate | @Roles on all endpoints; site-scoped queries via user.siteId |
| T-03-24 | Tampering | State machine bypass on tickets | mitigate | IncidentStateMachine.validateTransition() enforces valid transitions for MAINTENANCE_TICKET lifecycle (open → in_progress → resolved → closed) |
</threat_model>
