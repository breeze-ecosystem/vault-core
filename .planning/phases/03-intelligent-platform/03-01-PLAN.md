    Create `apps/api/migrations/timescaledb/up/014_analytics_aggregates.sql`:

    1. `zone_analytics_hourly` materialized view (continuous aggregate):
       ```sql
       CREATE MATERIALIZED VIEW zone_analytics_hourly
       WITH (timescaledb.continuous) AS
       SELECT
           time_bucket('1 hour', COALESCE(ae.time, dsl.time)) AS bucket,
           z.id AS zone_id,
           z.site_id,
           COUNT(*) FILTER (WHERE ae.decision = 'denied') AS denied_count,
           COUNT(*) FILTER (WHERE ae.decision = 'granted') AS granted_count,
           COUNT(*) FILTER (WHERE dsl.state IN ('forced', 'held-open')) AS door_anomaly_count,
           COUNT(*) FILTER (WHERE dsl.state = 'unsecured') AS unsecured_count,
           COUNT(DISTINCT ae.door_id) AS active_doors
       FROM zones z
       JOIN doors d ON d.zone_id = z.id
       LEFT JOIN access_events ae ON ae.door_id = d.id AND ae.time >= NOW() - INTERVAL '48 hours'
       LEFT JOIN door_state_log dsl ON dsl.door_id = d.id AND dsl.time >= NOW() - INTERVAL '48 hours'
       GROUP BY bucket, z.id, z.site_id
       WITH NO DATA;

       SELECT add_continuous_aggregate_policy('zone_analytics_hourly',
           start_offset => INTERVAL '3 days',
           end_offset => INTERVAL '1 hour',
           schedule_interval => INTERVAL '1 hour'
       );
       ```

    2. `site_analytics_daily` materialized view (continuous aggregate):
       ```sql
       CREATE MATERIALIZED VIEW site_analytics_daily
       WITH (timescaledb.continuous) AS
       SELECT
           time_bucket('1 day', COALESCE(ae.time, dsl.time, ie.time)) AS bucket,
           COALESCE(ae.site_id, dsl.site_id, ie.site_id) AS site_id,
           COUNT(*) FILTER (WHERE ae.decision = 'denied') AS total_denied,
           COUNT(*) FILTER (WHERE ae.decision = 'granted') AS total_granted,
           COUNT(DISTINCT dsl.door_id) FILTER (WHERE dsl.state IN ('forced', 'held-open')) AS doors_with_anomalies,
           COUNT(DISTINCT dsl.door_id) FILTER (WHERE dsl.state = 'unsecured') AS doors_unsecured,
           COUNT(DISTINCT ie.incident_id) AS incidents_created
       FROM access_events ae
       FULL JOIN door_state_log dsl ON dsl.site_id = ae.site_id AND dsl.time = ae.time
       FULL JOIN incident_events ie ON ie.site_id = COALESCE(ae.site_id, dsl.site_id) AND ie.time = COALESCE(ae.time, dsl.time)
       GROUP BY bucket, COALESCE(ae.site_id, dsl.site_id, ie.site_id)
       WITH NO DATA;

       SELECT add_continuous_aggregate_policy('site_analytics_daily',
           start_offset => INTERVAL '7 days',
           end_offset => INTERVAL '1 day',
           schedule_interval => INTERVAL '1 day'
       );
       ```

    3. Create indexes on the materialized views for efficient zone/site lookups:
       ```sql
       CREATE INDEX idx_zone_analytics_zone_bucket ON zone_analytics_hourly (zone_id, bucket DESC);
       CREATE INDEX idx_zone_analytics_site_bucket ON zone_analytics_hourly (site_id, bucket DESC);
       CREATE INDEX idx_site_analytics_site_bucket ON site_analytics_daily (site_id, bucket DESC);
       ```

    Follow the exact style from Phase 1's 004_continuous_aggregates.sql (CREATE MATERIALIZED VIEW WITH timescaledb.continuous, add_continuous_aggregate_policy, and indexes).

    **Part B — Analytics Service (analytics.service.ts):**

    Create `apps/api/src/modules/analytics/analytics.service.ts`. Inject `PrismaService`, `EventEmitter2`. No Redis needed (analytics queries hit TimescaleDB directly via continuous aggregates).

    Methods:

    1. `getZoneAnalytics(siteId?: string, zoneId?: string, from?: string, to?: string, granularity?: 'hourly' | 'daily')`:
       - Query `zone_analytics_hourly` (or _daily if granularity='daily') with WHERE filters, ordered by bucket DESC
       - JOIN to zones table for zone name
       - Return ZoneAnalyticsDto[]
       - If no continuous aggregate data (first run), fall back to raw event queries with LIMIT

    2. `getSiteAnalytics(siteId?: string, from?: string, to?: string)`:
       - Query `site_analytics_daily` with optional siteId filter, ordered by bucket DESC
       - JOIN to sites table for site name
       - Return SiteAnalyticsDto[]

    3. `getIntrusionEvents(siteId?: string, from?: string, to?: string)`:
       - Query from existing AI detection events / alerts. Intrusion detection already runs in Phase 1 AI pipeline — results are stored as Alert records with severity=CRITICAL and metadata containing intrusion zone info
       - Query: `SELECT a.id, a.camera_id, a.created_at, a.snapshot_url, a.metadata->>'zone_id' as zone_id, a.metadata->>'confidence' as confidence FROM alerts a WHERE a.severity = 'CRITICAL' AND a.metadata->>'type' = 'intrusion' AND ...`
       - Parse Alert metadata to extract zone_id, confidence
       - Return IntrusionEventDto[]
       - Implementation note: Alert.metadata is a Json field. Extract zone_id and confidence from metadata using PostgreSQL -> operator in the $queryRawUnsafe query

    4. `getLoiteringEvents(siteId?: string, from?: string, to?: string)`:
       - For MVP, detect loitering using SQL heuristic: access events where a door was held open for extended duration without a valid badge read
       - Query: `SELECT dsl.door_id, dsl.site_id, dsl.time as started_at, EXTRACT(EPOCH FROM (NOW() - dsl.time)) as duration_seconds, d.name as door_name, d.zone_id FROM door_state_log dsl JOIN doors d ON d.id = dsl.door_id WHERE dsl.state = 'held-open' AND dsl.time > ...`
       - Group by door_id, use duration threshold (configurable, default 300 seconds = 5 min)
       - Return LoiteringEventDto[]

    5. `getUnusualAbsence(siteId?: string, zoneId?: string)`:
       - Detect zones that are expected to have activity (based on schedule) but show no recent access events
       - Query: Check `zone_analytics_hourly` for zones with granted_count = 0 in the last hour when schedules indicate active hours
       - For MVP: simpler — detect zones with doors that have schedules (day-of-week + time-of-day matching) but zero access events in the last 2 hours during scheduled active time
       - Return AbnormalActivityDto[] with metric='unusual_absence'

    6. `getAbnormalActivity(siteId?: string, zoneId?: string)`:
       - Compute historical baseline from `zone_analytics_hourly` over last 7 days (same hour of day, same day of week)
       - Compare current hour's metrics to baseline
       - Calculate z-score: (current_value - mean) / stddev
       - Flag as abnormal if z-score > 2.0 (configurable threshold)
       - Return AbnormalActivityDto[] with metric='abnormal_activity' for each zone where current metric deviates significantly from baseline
       - Implementation: For MVP, compute baseline by averaging denied_count and door_anomaly_count per zone for same day-of-week + hour in the last 7 days. Compare current hour's values.

    7. `getAnalyticsTrends(siteId: string, metric: string, granularity?: 'hourly' | 'daily')`:
       - Return time-series data points for trend chart rendering
       - Query aggregate table for the given metric over time
       - Return AnalyticsTrendPoint[]
