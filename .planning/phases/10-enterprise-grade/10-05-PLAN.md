
2. **Add to @Module imports array** — Insert new modules alphabetically in the imports array. The full module order (alphabetical): AccessModule → AiAgentModule → AiModule → AlertModule → AnalyticsModule → AnprModule → **ApiKeyModule** → AuditModule → AuthModule → CameraModule → ChatModule → **ComplianceModule** → ContactModule → CorrelationModule → DashboardModule → DoorModule → EquipmentModule → FeatureGateModule → GovernanceModule → IncidentModule → InferenceModule → IngestionModule → InviteModule → LicenseModule → MaintenanceModule → MqttModule → NotificationModule → NotificationsModule → OrganizationModule → PatternsModule → PrismaModule → RiskModule → **SsoModule** → SupervisionModule → UserModule → VisitorModule → **WebhookModule**

3. **Verify guard order**: The existing guard chain is JwtAuthGuard → TenantIsolationGuard → RolesGuard → FeatureGateGuard. The TenantApiKeyGuard is NOT registered as a global APP_GUARD — it's applied per-controller via @UseGuards() on the v1 controllers. This avoids conflict with JWT-based routes.

4. **Rate limiter conflict resolution** (Pitfall 5): The global @fastify/rate-limit protects `/api/*` routes. The v1 endpoints `/api/v1/*` use per-key rate limiting in TenantApiKeyGuard instead. Add a comment documenting that if the global rate limiter causes double-counting, it should be configured to exclude `/api/v1/*` paths. For now, the per-key guard runs inside the NestJS request lifecycle and the global Fastify rate limiter runs at the HTTP layer — they may both trigger. This is acceptable for initial delivery; the exclusion hook can be added in a follow-up.

5. **No middleware changes needed** — TenantContextMiddleware already applies to all routes (`forRoutes('*')`). SSO callback routes are @Public() so they bypass JwtAuthGuard but still go through TenantContextMiddleware (which needs an orgId — for SSO callbacks, the orgId is passed in the relay state or derived from the IdP config lookup).

**IMPORTANT: Handle SSO callback org context.** For SSO callback routes (POST /api/auth/sso/saml/callback, GET /api/auth/sso/oidc/callback), the request has no JWT and thus no orgId. The TenantContextMiddleware currently destroys connections without orgId. SSO callbacks must be exempted — the SsoService resolves orgId from the IdP config during SAML assertion validation. Ensure SSO callback routes work by having SsoService set the org context in AsyncLocalStorage manually during validate(). Add a try/catch around the SSO callback's org context setup: in SsoController's callback methods, after Passport validation, call `orgContext.run({ orgId: user.orgId }, () => { /* token exchange */ })`.
</action>

Add typed fetchWithAuth functions for all new Phase 10 endpoints in `apps/dashboard/lib/api.ts`. Follow the existing pattern (export async function, fetchWithAuth, typed response). Functions to add:

**SSO Configuration:**
- `fetchIdpConfig()` → GET /api/auth/sso/config → returns `IdpConfig | null`
- `saveIdpConfig(data: CreateIdpConfigInput)` → POST /api/auth/sso/config → returns `IdpConfig`

**API Keys:**
- `fetchApiKeys()` → GET /api/api-keys → returns `TenantApiKey[]` (masked, no keyHash)
- `createApiKey(data: CreateTenantApiKeyInput)` → POST /api/api-keys → returns `{ id, name, keyPrefix, scopes, rateLimit, rawKey }` (rawKey returned once)
- `revokeApiKey(id: string)` → DELETE /api/api-keys/:id → returns void
- `fetchApiKeyUsage(id: string)` → GET /api/api-keys/:id/usage → returns `{ lastUsedAt, recentRequestCount }`

**Webhooks:**
- `fetchWebhookSubscriptions()` → GET /api/webhooks/subscriptions → returns `WebhookSubscription[]`
- `createWebhookSubscription(data: CreateWebhookSubscriptionInput)` → POST /api/webhooks/subscriptions → returns `WebhookSubscription`
- `updateWebhookSubscription(id: string, data: UpdateWebhookSubscriptionInput)` → PATCH /api/webhooks/subscriptions/:id → returns `WebhookSubscription`
- `deleteWebhookSubscription(id: string)` → DELETE /api/webhooks/subscriptions/:id → returns void
- `fetchWebhookDeliveries(id: string, filters?: { status?: string; from?: string; to?: string })` → GET /api/webhooks/subscriptions/:id/deliveries → returns `WebhookDelivery[]`
- `retryWebhookDelivery(subscriptionId: string, deliveryId: string)` → POST /api/webhooks/subscriptions/:subscriptionId/retry/:deliveryId → returns void

**Compliance:**
- `generateComplianceReport(data: GenerateComplianceReportInput)` → POST /api/compliance/reports/generate → returns blob (set responseType to 'blob', trigger download)

**Organization Branding:**
- `fetchOrganizationBranding()` → GET /api/organizations/me → returns `{ logoUrl, primaryColor, displayName }` (extract from org response or dedicated endpoint)
- `updateOrganizationBranding(data: { logoUrl?: string; primaryColor?: string; displayName?: string })` → PATCH /api/organizations/branding → returns `Organization`

**Import types from @repo/shared** where available (CreateIdpConfigInput, CreateTenantApiKeyInput, CreateWebhookSubscriptionInput, UpdateWebhookSubscriptionInput, GenerateComplianceReportInput). For types not yet in shared package, define inline TypeScript interfaces in api.ts following existing patterns.

Place new functions after existing API functions. Group with section comments matching the existing pattern (e.g., `// SSO Configuration`, `// API Keys`, `// Webhooks`, `// Compliance`, `// Branding`).
</action>
