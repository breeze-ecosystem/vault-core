**`apps/dashboard/components/command-center/CommandCenterFeed.tsx`** — New component:

A vertically scrolling real-time event feed component that connects to multiple WebSocket namespaces and aggregates events into a single chronological stream.

- **Socket.IO connections**: Connect to existing WebSocket namespaces that emit events:
  - `/ws/doors` — door state changes (existing from Phase 1)
  - `/ws/alerts` — new alert notifications 
  - `/ws/webhooks` — webhook delivery status (from Plan 03 webhook gateway)
  - `/ws/incidents` — incident status updates
  
  For namespaces that don't exist yet (alerts, incidents), create a simple event-based approach: listen for `@OnEvent` in existing modules and emit via a new unified command-center gateway. For MVP, connect to the door gateway and webhook gateway (both exist), and poll for alerts/incidents.

- **Unified feed approach**: Each event is normalized into a FeedEvent interface:
  ```typescript
  interface FeedEvent {
    id: string;
    type: "door" | "alert" | "incident" | "webhook" | "ai";
    severity: "critical" | "high" | "medium" | "low" | "info";
    title: string;
    description: string;
    time: Date;
    source: string; // e.g., "Porte Entrée Principale", "Caméra Parking Nord"
  }
  ```

- **Component structure**:
  - State: `events: FeedEvent[]` (capped at last 100)
  - Connect to Socket.IO on mount using `socket.io-client` (already installed in dashboard per STACK.md)
  - On each socket event, normalize and prepend to events array
  - UI: Virtualized list of events (use `scroll-area` component). Each event row: colored severity indicator left border, icon per type, title, description, relative timestamp, source badge
  - Filter bar at top: filter by event type (door/alert/incident/ai) and severity
  - Sound notification for critical events (optional — toggle in settings)
  - Auto-scroll to top on new event
  - Max 100 events in memory (older events dropped)

- **Connection management**: Reconnect on disconnect with exponential backoff. Show "Déconnecté" banner when offline. Auth token from useAuth() passed as `auth: { token }` in socket handshake.

```typescript
export interface PendingDoorAction {
  id: string;
  doorId: string;
  action: "open" | "close" | "lock";
  queuedAt: number;
}

export async function queueDoorAction(action: Omit<PendingDoorAction, "id" | "queuedAt">): Promise<void> {
  const pending = await getPendingDoorActions();
  pending.push({ ...action, id: Date.now().toString(), queuedAt: Date.now() });
  await AsyncStorage.setItem(DOOR_ACTIONS_KEY, JSON.stringify(pending));
}

export async function getPendingDoorActions(): Promise<PendingDoorAction[]> {
  const raw = await AsyncStorage.getItem(DOOR_ACTIONS_KEY);
  return raw ? JSON.parse(raw) : [];
}

export async function clearPendingDoorActions(): Promise<void> {
  await AsyncStorage.removeItem(DOOR_ACTIONS_KEY);
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Mobile device → API | Guard mobile app authenticates via JWT (existing auth context); NFC/QR/door control API calls cross trust boundary |
| Dashboard → WebSocket | Socket.IO connection from dashboard to API for real-time feed — must validate org context in handshake |
| NFC tag → Mobile | Physical NFC badge data is untrusted input — must validate credential against database, not trust tag data directly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-31 | Spoofing | NfcScanner — badge validation | mitigate | Badge ID validated server-side against Credential table (never trust client-provided badge data); server returns actual credential status |
| T-10-32 | Information Disclosure | CommandCenterFeed — WebSocket | mitigate | Socket.IO handshake validates auth token (JWT); client only receives events for its org (server-side room scoping) |
