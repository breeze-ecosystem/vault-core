       - @Injectable() with Logger
       - Inject PrismaService, AuditService, ConfigService
       - Private otpStore: Map<string, { code: string; expiresAt: Date; attempts: number }>
       - Method `async requestOtp(email: string): Promise<{ success: boolean; message: string }>`:
         - Rate limit: check existing OTP for this email — if within 60 seconds, reject with "Veuillez attendre avant de demander un nouveau code"
         - Generate 6-digit random code (Math.floor(100000 + Math.random() * 900000).toString())
         - Store in otpStore with 15-minute TTL and attempts counter (0)
         - Log to audit: action "SUBJECT_ACCESS_OTP_REQUESTED"
         - Return success (do NOT return the code in response — it's sent via email)
         - NOTE: For v1, also log the code to console for development/testing (remove before production)
       - Method `async verifyOtp(email: string, code: string): Promise<SubjectDataDto>`:
         - Look up OTP in store
         - If not found or expired: throw BadRequestException("Code invalide ou expiré")
         - If attempts >= 3: throw BadRequestException("Trop de tentatives. Demandez un nouveau code.")
         - Increment attempts
         - If code matches: clear OTP from store
           - Gather subject data:
             - Find user by email in OrganizationMember/User records
             - Find related Face records (by name match)
             - Find related Alert/Incident records (by user association)
             - Return SubjectDataDto with email, name, role, sites, relatedAlerts count, relatedFaces count, dataItems array
           - Log to audit: action "SUBJECT_ACCESS_DATA_VIEWED"
         - If code doesn't match: throw BadRequestException("Code incorrect")
       - Method `async submitRequest(email: string, type: "rectify" | "delete", details: string): Promise<{ referenceId: string }>`:
         - Create SubjectAccessRequest record in Prisma with status "PENDING"
         - Reference ID format: `SAR-{shortId}-{timestamp}`
         - Log to audit: action "SUBJECT_ACCESS_REQUEST_SUBMITTED"
         - Dispatch webhook: BASTION_EVENT_TYPES.SUBJECT_ACCESS_REQUEST
         - Return { referenceId }
       - Private method `cleanupExpiredOtps()`: periodically remove expired OTPs from map to prevent memory leak

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-11 | Information Disclosure | Subject access — unauthorized data access | mitigate | OTP verification via email + 6-digit code. 3-attempt limit. 15-min TTL. Rate-limited per email. OTP never returned in response. |
| T-04-12 | Spoofing | Subject access — fake identity verification | mitigate | OTP sent to email on file. No bypass without email access. All attempts logged in audit hash-chain. |
| T-04-13 | Tampering | Subject data deletion — abuse prevention | mitigate | Rectify/delete requests are PENDING until admin approval. Audit trail logs all requests. Admin workflow for approval. |
| T-04-14 | Information Disclosure | HAPDP declaration — org data exposure | mitigate | All HAPDP endpoints require @Roles("ADMIN", "SUPER_ADMIN") + @RequiresPack("BASTION"). TenantIsolationGuard enforces org scoping. |
| T-04-15 | Repudiation | Missing audit on subject data access | mitigate | All subject access operations logged via AuditService with SHA-256 hash-chain. @Audited() decorator on all HAPDP mutation endpoints. |
| T-04-16 | Denial of Service | OTP brute force via public endpoint | mitigate | @fastify/rate-limit globally configured. Service-level rate limit: 3 attempts per email per 15-min window. OTP map cleanup periodic. |
